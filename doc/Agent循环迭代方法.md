# ProtoVault Agent 循环迭代方法

## 1. 对 LoopAgent 的理解

这里的 LoopAgent 不是“让 Agent 无限自由发挥”，而是一种有边界的循环工作法。

它可以分成两层：

1. 工作流层：把一组步骤放入循环中，直到达到最大轮次、通过验收或触发退出条件。
2. 产品工程层：把目标、状态、验收门、变更记录和下一轮任务都显式写下来，让 Agent 可以长时间推进，但每一轮都能恢复、复盘和审计。

对 ProtoVault 来说，更适合采用第二层方法：不引入新的运行时框架，而是把现有开发计划、测试脚本、更新日志和 Git 提交组织成可重复执行的产品迭代闭环。

参考来源：

- [Google ADK 的 Loop Agents 文档](https://adk.dev/agents/workflow-agents/loop-agents/)：强调循环工作流、最大迭代次数和显式退出条件。
- [Addy Osmani 的 loop engineering 文章](https://addyosmani.com/blog/loop-engineering/)：强调长时间编码 agent 需要目标、状态、评估、反馈和人类检查点。

## 2. ProtoVault 的循环单元

每一轮 Agent 循环只处理一个明确目标，目标大小以“可以在一次测试和一次提交中验收”为宜。

推荐循环单元：

```text
选择目标
→ 记录当前状态
→ 明确验收门
→ 实现或设计收束
→ 运行检查
→ Review 风险
→ 更新文档与日志
→ 提交 Git
→ 写入下一轮候选任务
```

循环的关键不是“做更多”，而是“每轮结束后，下一次能从事实状态继续”。

## 3. 本地命令

当前项目提供三个入口：

```powershell
pnpm agent:loop:status
pnpm agent:loop
pnpm agent:loop:release
```

- `agent:loop:status`：只打印当前 Git 状态、最新提交和安全警告。
- `agent:loop`：运行轻量检查，适合普通功能迭代后的快速闭环。
- `agent:loop:release`：运行完整发布门，适合阶段验收或高风险改动。

脚本会显式提示 `examples/` 下的未归属改动和临时文件，避免长循环误提交用户试验文件。

## 4. 每轮记录模板

每轮结束时应更新 `doc/Agent循环运行记录.md`：

```text
## YYYY-MM-DD Loop N：目标名称

### 目标

本轮要完成什么，以及为什么现在做。

### 基线状态

- 当前阶段。
- Git 状态。
- 已知脏文件。
- 不归本轮处理的内容。

### 行动

- 关键实现或设计变更。
- 涉及文件。

### 验证

- 运行的命令。
- 结果。

### 风险与限制

- 哪些地方仍不可靠。
- 哪些能力只是 MVP。

### 下一轮

- 下一轮最应该处理的 1 到 3 个目标。
```

## 5. 验收门分级

### 轻量门

适合小型前端、契约或文档改动：

```powershell
pnpm agent:loop
```

当前包含：

- contracts 单元测试。
- desktop TypeScript 类型检查。
- desktop 单元测试。

### 发布门

适合阶段完成、跨层改动、同步/生成/解析等高风险改动：

```powershell
pnpm agent:loop:release
```

当前等价于：

```powershell
pnpm release:check
```

## 6. 安全边界

长循环工作不能扩大授权边界：

- 不自动删除用户文件。
- 不自动覆盖未归属脏改动。
- 不把临时文件、演示试验文件混入提交。
- 不在解析失败时继续写盘覆盖 Header。
- 不把“没有报错”当作“用户体验正确”，每轮都应包含一次人工习惯视角的 review。

## 7. 对下一阶段的应用

P13 网络派生分析很适合使用循环推进，因为它既有模型、UI、派生算法，也有文档和测试。

建议拆成以下循环：

1. FlowView 配置保存与过滤。
2. 节点收发汇总和链路负载排序。
3. 关键链路/瓶颈提示。
4. 网络地图报告生成。
5. 与语义 Diff、协议文档生成联动。

每一轮都必须以“用户能看到什么新信息、能做出什么更好判断”为验收标准，而不只是新增字段或表格。
