# ProtoVault 产品审核与架构升级建议（2026-07-10）

## 1. 审核结论

ProtoVault 已经形成可运行的 Windows 桌面 MVP：能够扫描 C++ Header、展示和编辑协议、分析内存布局、同步注释、执行 Lint 和语义 Diff，并提供协议网络地图、Git Source Control、本地 AI 使用助手和 Windows 安装包。

当前主要问题已经从“功能缺失”转为“核心模型和工程边界不够稳固”。产品功能宽度走在了底层架构前面：界面可以完成大量操作，但实际运行仍以 Electron 主进程中的 Node/TypeScript 单体服务和 Clang CLI 为核心；最初规划的统一协议 IR、C++ 协议服务、SQLite 增量索引、稳定对象身份和完整冲突中心尚未真正成为主链路。

因此，下一版不建议继续横向增加新视图。优先级应切换为：

```text
稳定身份
→ 统一 IR
→ 持久增量索引
→ 可取消的解析任务
→ 前端模块化
→ 完整冲突与版本工作流
```

产品定位建议从“数据协议管理工具”升级为：

> 面向 C++ 数据协议、系统通信网络与版本治理的本地工程工作台。

## 2. 最初设计与当前事实对照

| 主题 | 最初设计 | 当前事实 | 审核判断 |
|---|---|---|---|
| 桌面框架 | Electron + React + TypeScript | 已实现并可打包 | 达标 |
| 本地协议服务 | C++20 + libclang，业务接口独立于 Electron | C++ 服务仍是 health/self-test 骨架；主要业务在 Electron 主进程 `workspace.ts`，调用 `clang++` CLI | 核心架构未完成 |
| 统一协议 IR | TS/C++ 无损往返，所有功能基于同一 IR | Contracts 中存在规范 IR，运行界面使用另一套 `WorkspaceView`；字段类型仍以字符串为主 | 双模型并存，需要收敛 |
| 稳定 ID | 改名不应被识别为删除后新增 | 类型 ID 由完全限定名哈希生成，字段 ID 由类型名和字段名生成 | 不满足原始目标，是 P10 的实质缺口 |
| 工作区索引 | `.protocol` + SQLite 增量索引 | 有工作区记录、元数据和网络 JSON；没有 SQLite 持久索引 | MVP 可用，大项目能力不足 |
| Header 解析 | libclang/AST、有限语法范围、明确诊断 | Clang JSON AST + 受限候选过滤，支持常见 Header；复杂宏/模板排除 | MVP 达标，进程模型待升级 |
| 布局分析 | 与编译器 `sizeof/offsetof` 对照 | 支持范围已有编译器交叉测试 | MVP 达标；结果仍是估算源 |
| 双向同步 | 最后有效 IR、哈希基线、冲突选择 | 有 hash、防覆盖、外部修改提醒和源码修复；本轮补充会话内最后有效 IR | 持久最后有效 IR、三方冲突解决仍未完成 |
| 代码生成 | 确定性 IR → Header | 当前以受控文本片段和源码重写为主 | 纵切完成，完整生成器未完成 |
| 元数据 | 描述、单位、范围、坐标系、时间基准等 | 注释与部分网络元数据已实现；单位、范围等语义属性未形成主界面工作流 | 部分完成 |
| Git 版本治理 | Branch/Tag/Baseline/Semantic Diff | 本地 Source Control、提交 Graph、Diff、分支和 Tag 已实现 | 本地闭环达标；远端和多协议包待完成 |
| 协议网络 | 节点、链路、协议绑定为事实，数据流为派生视图 | 已实现并完成基本 CRUD、带宽估算、FlowView 和报告 | 模型方向正确；缺真实运行数据 |
| 本地 AI | 本地模型辅助 | Ollama 模块化问答与离线降级已实现 | 使用助手可用，尚不是协议智能分析 |
| 发布 | Windows 安装包 | Setup/Portable 已生成，Git/Clang 随包集成 | 初始发布达标；体积、签名、更新待完成 |

## 3. 当前产品值得保留的设计

### 3.1 三层事实模型

当前最有价值的设计不是某个图表，而是已经形成的三层事实：

1. 源码与 Git：回答“文件现在是什么、从哪里演进而来”。
2. 协议语义：回答“数据结构、字段、布局和兼容性是什么”。
3. 系统网络：回答“哪些节点通过哪些链路传输哪些协议”。

FlowView、瓶颈提示、影响分析和未来 AI 结论都应该是以上事实的派生结果，不再成为新的第一层编辑数据。

### 3.2 中间编辑、右侧只读 Inspector

用户高频编辑继续集中在中间 tab：字段表格、枚举表格和源码编辑器。右侧 Inspector 用于大小、偏移、风险和关系上下文，不再承担旧的数据流标签编辑。这一规则应保持一致。

### 3.3 Git Tag 作为协议基线

以 Git 分支表达演化线、Tag 表达可交付基线，比自建 Snapshot 更容易审计和复现。`.protocol` 保存的是语义补充、索引和报告，不再复制一套隐式版本历史。

## 4. 代码与性能审核

### 4.1 已在本轮直接修复

- Header 文件读取由串行改为有界并行。
- 新增 include 依赖闭包指纹和进程内 AST 结果缓存；修改被 include 的 Header 会使依赖方缓存失效。
- Header 解析失败时保留当前会话中该文件的最后有效类型，源码 tab 仍可修复。
- 保存前语法检查的 include 参数从“所有目录”收敛到 Header 父链目录。
- Git 仓库上下文复用；`status` 内部查询改为并行。
- Git Graph 从“每个提交额外运行一次 `diff-tree`”改为一次 `git log --name-status` 读取，消除 N+1 子进程。
- Git 输出缓冲上限提升并关闭 Unicode 路径转义。
- 问题面板改为分组、限量、简明信息和源码跳转，避免长 Clang 命令撑满 Inspector。
- 全局快捷键监听改为稳定注册；F2/Alt 导航不再抢占文本输入。
- Escape 可关闭应用内确认/输入弹窗。
- 三栏宽度和折叠状态写入本地偏好，重启后恢复。
- 扫描完成条自动收起。
- 移除旧 Struct producer/consumer 的不可达前端状态、保存分支和数据流图谱渲染代码。

### 4.2 仍存在的主要代码风险

1. `renderer/src/main.tsx` 仍超过 6000 行，承担应用状态、业务命令、树、编辑器、网络、Git、图谱和 Assistant。生产 bundle 约 952 KB，尚未按视图拆包。
2. `main/workspace.ts` 超过 3000 行，文件发现、Clang、布局、元数据、网络、Git、Lint、报告、Diff 和写盘集中在同一模块。
3. 解析一个 Header 时，每个候选 struct/enum 可能独立启动一次 Clang；类型密集 Header 的冷扫描成本仍高。
4. AST 缓存当前只在进程内有效，应用重启后仍需冷扫描。
5. 扫描任务没有取消令牌、代次控制和超时分类；快速切换工作区时缺少明确的旧任务丢弃协议。
6. Git/Clang 错误仍以普通 `Error` 为主，尚未统一成可定位、可恢复、可建议操作的业务错误模型。
7. 当前源代码编辑器是自研高亮文本区，不具备 Monaco/CodeMirror 的完整诊断、撤销栈和大型文件编辑能力。

## 5. 下一版产品架构

### 5.1 明确三个核心域

```text
Protocol Workspace
├─ Source Domain：Header、Include、Git、文件冲突
├─ Protocol Domain：IR、布局、语义元数据、Lint、Diff
└─ Network Domain：节点、链路、协议绑定、FlowView、运行指标
```

模块间只通过带版本的契约通信。UI 不直接依赖 Clang AST 节点，也不直接用 Header 文本正则推导业务关系。

### 5.2 推荐的运行架构

- Electron 主进程继续负责窗口、文件对话框、文件监听、Git 和任务调度。
- 独立 Parser Worker 负责 Clang 解析与布局探测。短期可先使用 Node worker/child process 封装现有 CLI；中期再把解析核心迁入 C++20/libclang 服务。
- SQLite 保存文件指纹、include 图、IR 索引、诊断和版本迁移信息；`.protocol/*.json` 继续保存适合审计和版本控制的用户数据。
- React 端按 Explorer、Editor、Network、Source Control、Graph、Assistant 拆分视图包，并建立统一 command/navigation/dirty-document 服务。

不建议把 Git、文件监听和全部业务一次性迁入 C++。C++ 服务首先只接管高成本、强类型的解析和 ABI 能力，迁移收益更明确。

### 5.3 稳定身份方案

下一版必须引入持久对象身份表，例如 `.protocol/identity.json` 或 SQLite identity 表：

- 首次导入时为 Header、Struct、Enum、Field 分配 UUID。
- 保存源码时通过文件、父对象、字段邻接、类型签名和源码范围做重关联。
- 明确改名操作直接沿用原 ID。
- 外部改名由匹配评分产生“可能改名”候选，低置信度时交给用户确认。
- 语义 Diff 只在身份确实断裂时报告删除/新增。

这是版本治理、元数据保留和影响分析可信度的共同前提。

## 6. P18 建议执行计划

| Loop | 目标 | 主要交付 | 验收门 |
|---|---|---|---|
| P18-0 | 模型收敛 ADR | 定义 Canonical IR、Workspace Projection、Network Model 和迁移边界 | 新功能不得直接扩展第二套类型模型 |
| P18-1 | 持久增量索引 | SQLite 文件指纹、include 图、Header IR cache、schema migration | 重启后 warm scan；单文件修改只解析依赖闭包 |
| P18-2 | 稳定 ID | identity store、显式改名继承、外部改名候选 | 类型和字段改名不再被 Diff 判为删除+新增 |
| P18-3 | Parser Worker | 任务 ID、取消、超时、进度、崩溃隔离 | 切换工作区不会被旧扫描覆盖；解析崩溃不影响主进程 |
| P18-4 | 前端模块化 | 拆分 App shell、Explorer、Protocol Editor、Network、Git、Graph | 主入口显著缩小；视图按需加载；现有 E2E 不回退 |
| P18-5 | 冲突中心 | Disk/Base/Draft 三方状态、重新导入、覆盖、手工合并 | 外部修改和本地草稿不会静默丢失 |
| P18-6 | 版本资产 | Remote fetch/push、分支模板、多协议包挂载与只读合成预览 | 导出不覆盖源目录；冲突有报告 |
| P18-7 | 发布硬化 | 安装体积、正式图标、签名、升级与崩溃日志 | 干净 Windows VM 可安装、升级、回滚并运行 |

## 7. 建议性能预算

以下数字应作为测试目标，不作为当前已达成承诺：

- 500 个 Header 的 warm reopen：3 秒内展示可用索引。
- 单 Header 保存：1 秒内完成文本校验反馈，依赖重解析异步推进。
- 扫描进度更新：每 100 ms 最多一次，避免 IPC 事件风暴。
- Git Graph 100 个提交：500 ms 内展示首屏。
- 协议树 10,000 节点：搜索输入保持可交互，必要时使用虚拟列表。
- 关系图谱 2,000 节点：WebGL 路径下保持 45 FPS 以上。

## 8. 需要共同确认的四个决策

1. 是否接受 P18 暂停新增横向功能，先完成稳定 ID、持久索引和任务隔离。
2. C++ 服务是否只先接管 Parser/ABI，而 Git、文件系统和网络模型继续留在 Node 主进程。
3. 源码编辑器是否引入 Monaco；它会增加包体，但能换来标准诊断、撤销、搜索和大型文件体验。
4. 多协议包合成的首版是否严格只读预览和导出，不直接执行 Git merge 或覆盖源 Header。

本审核建议的默认答案依次是：接受、接受、引入 Monaco、首版只读。
