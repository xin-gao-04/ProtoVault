# ProtoVault 关系图谱最终设计

本文档用于收敛 ProtoVault 的关系图谱设计。目标不是做一个“看起来很酷的 Obsidian 克隆”，而是把图谱做成协议管理工作中的影响分析入口。

## 1. 设计结论

关系图谱最终定位为：

```text
Protocol Impact Graph
协议影响图谱
```

它的核心价值是回答四个问题：

1. 当前协议类型依赖谁？
2. 如果修改当前类型，会影响谁？
3. 哪些类型存在布局、诊断或元数据风险？
4. 如何从全局关系快速跳回左侧树、表格编辑和 Header 源码？

因此，图谱不应成为一个独立炫技视图，而应成为“扫描—理解—编辑—检查—Diff”闭环中的导航和风险解释层。

## 2. 真实使用场景

### 场景 A：接手一个陌生 Header 工作区

用户打开一个包含多个 Header 的目录后，需要快速理解：

- 哪些结构体是核心协议。
- 哪些类型只是底层基础类型。
- 哪些 Header 声明密集、耦合较高。

图谱应该默认展示全局依赖关系，但不能把所有字段、枚举值、include 细节全部铺开。否则用户看到的是噪声星空，而不是协议地图。

最终设计：

- 默认节点：Struct / Enum。
- Header 只作为分组或弱背景，不作为主要认知对象。
- 字段不作为默认节点，只作为边的来源信息。
- 节点大小表示影响力，不表示单纯字段数量。

### 场景 B：修改某个结构体前评估影响

用户准备修改字段类型、删除字段、调整字段顺序时，最关心的是：

- 谁引用了这个结构体？
- 影响范围是一跳还是多跳？
- 会不会导致布局变化扩散？

最终设计：

- 单击节点进入聚焦态。
- 聚焦态默认展示两层关系：
  - 本类型依赖的类型。
  - 依赖本类型的类型。
- 无关节点降透明，不直接隐藏，避免用户迷失上下文。
- 右侧属性栏显示影响摘要。

### 场景 C：检查 ABI 和内存布局风险

ProtoVault 不是普通 Header 编辑器，内存布局是核心价值。图谱应该把 P4 布局分析结果变成直观信号：

- 哪些结构体 padding 高。
- 哪些结构体 size/align 敏感。
- 哪些结构体被大量引用，改动风险高。

最终设计：

- 节点外环表示布局风险。
- 外环颜色：
  - 绿色：布局稳定。
  - 黄色：存在 padding 或对齐关注点。
  - 红色：布局变化、异常 pack、或高风险诊断。
- 悬停卡片显示 size、align、padding、字段数、引用数。

### 场景 D：处理诊断与 Lint

用户不应该在图谱里完整阅读问题列表，但图谱应该告诉用户“哪里值得点进去处理”。

最终设计：

- 节点角标显示诊断数量。
- 严重错误节点优先高亮。
- 点击节点后右侧属性栏展示该节点相关诊断摘要。
- 具体修复仍回到中间表格、Header 源码和问题面板完成。

### 场景 E：查看 Diff 影响

Diff 图谱很有价值，但不应该做成一个和普通图谱完全分离的复杂模式。它应该是普通图谱上的临时叠加层。

最终设计：

- 当前工作区有快照 Diff 时，图谱可开启“变更叠加”。
- 新增节点：绿色边框。
- 删除节点：红色虚线幽灵节点。
- 修改节点：黄色边框。
- Breaking change：红色高亮。
- offset / size 改变在悬停卡片和右侧详情中展示。

## 3. 应该减少或取消的功能

以下功能看起来有吸引力，但对 MVP 协议管理帮助有限，应延后或取消。

### 3.1 不做默认真 3D WebGL

原因：

- 真 3D 增加 Three.js/WebGL 依赖和调试成本。
- 协议依赖图的关键是可读性，不是空间沉浸感。
- 3D 容易遮挡标签和边，反而降低工程分析效率。

最终取舍：

- MVP 保留 Canvas 2.5D 动效。
- 使用轻微 z 深度、发光、漂浮和缩放制造空间感。
- 如果后续项目规模大、用户确认需要，再替换为 WebGL 渲染器。

### 3.2 不把字段默认做成节点

原因：

- 字段数量通常远大于类型数量。
- 默认展示字段会导致图谱爆炸。
- 用户真正关心的是类型之间的影响链。

最终取舍：

- 字段作为边的解释信息。
- 悬停边或节点时显示相关字段。
- 只在选中某个 Struct 后，在右侧详情或中间表格展示字段级信息。

### 3.3 不做过多模式切换

之前设想过依赖、布局、Lint、Diff、元数据五个模式。这个设计过重。

最终取舍：

只保留一个主图谱和三个轻量叠加：

```text
主图谱：类型依赖关系
叠加 1：布局风险
叠加 2：诊断风险
叠加 3：Diff 变化
```

元数据完整性不单独做成图谱模式，只作为节点角标和右侧详情中的质量信息。

### 3.4 不在图谱中做编辑

原因：

- 图谱编辑容易造成误操作。
- 当前项目已经明确“所有编辑尽量在中间表格完成”。
- 图谱更适合查看、定位和影响解释。

最终取舍：

- 图谱只读。
- 双击打开 tab。
- Ctrl/快捷入口跳转到表格或源码编辑。
- 编辑后图谱根据最新 IR 刷新。

### 3.5 不展示所有 include 关系

原因：

- include 图通常比协议语义依赖更噪声。
- include 不等于协议引用。

最终取舍：

- 默认隐藏 include 边。
- Header 作为分组或背景区域。
- 未来可在调试入口临时开启 include 关系，但不作为主体验。

## 4. 最终信息架构

### 4.1 节点

默认节点：

| 节点 | 是否默认显示 | 说明 |
|---|---:|---|
| Struct | 是 | 图谱主节点 |
| Enum | 是 | 图谱主节点 |
| Header | 弱显示 | 作为分组、背景或节点索引信息 |
| Field | 否 | 作为边和详情信息 |
| Enum Value | 否 | 仅在详情中展示 |
| Diagnostic | 否 | 作为角标和详情摘要 |

### 4.2 边

默认边：

| 边 | 示例 | 是否默认显示 | 说明 |
|---|---|---:|---|
| 类型引用 | `RadarTrack -> Timestamp` | 是 | 字段类型引用 |
| Header 声明 | `track.hpp declares RadarTrack` | 弱显示 | 用于分组和来源解释 |
| Include | `track.hpp includes time.hpp` | 否 | 过于噪声，默认隐藏 |
| Diff 影响 | `Changed Timestamp affects RadarTrack` | 叠加 | 仅在 Diff 场景显示 |

### 4.3 视觉编码

默认编码应该简单稳定，避免让用户学习太多图例。

| 视觉元素 | 表示含义 |
|---|---|
| 节点大小 | 影响力：入度、出度、被引用次数综合 |
| 节点颜色 | 类型类别：Struct / Enum |
| 节点外环 | 风险等级：布局、诊断、Diff 综合 |
| 节点角标 | 诊断数量或未补全元数据数量 |
| 边粗细 | 引用强度，通常由字段引用数量决定 |
| 边方向 | 从使用者指向被引用类型 |
| 透明度 | 与当前聚焦节点的相关性 |

## 5. 布局与界面设计

### 5.1 页面结构

```text
左侧 rail
  ├─ 工作区
  ├─ 关系图谱
  └─ 问题/后续扩展

左侧树
  └─ 保持文件树 + 协议树结构

中间主视图
  ├─ 顶部图谱标题栏
  ├─ Canvas 图谱区域
  ├─ 图谱工具条
  └─ 节点索引 / 搜索结果

右侧属性栏
  ├─ 当前节点摘要
  ├─ 影响范围
  ├─ 布局风险
  ├─ 诊断摘要
  └─ 快捷跳转
```

### 5.2 中间图谱区域

中间区域应尽量接近 Obsidian 图谱体验：

- 深色背景。
- 节点细小但清晰。
- 标签按优先级显示，不全部常驻。
- 边细且低对比，聚焦后增强。
- 支持拖拽节点、拖动画布、滚轮缩放。

但需要比 Obsidian 更工程化：

- 悬停卡片必须显示协议属性。
- 聚焦节点必须显示影响链。
- 右侧必须给出可执行摘要。

### 5.3 右侧属性栏

当处于图谱视图时，右侧属性栏不要只显示普通文件属性，而要切换为图谱上下文。

选中节点后显示：

```text
名称：RadarTrack
类型：Struct
来源：headers/radar/track.hpp
Size：128
Align：8
Padding：12
字段数：9
引用了：Timestamp、Vec3、TrackState
被引用：TrackCluster、TelemetryFrame
诊断：1 warning
元数据：缺少 2 个字段注释
操作：打开 tab / 定位树 / 查看源码
```

未选中节点时显示：

```text
类型数
引用边数
高风险节点数
诊断节点数
最大影响节点 Top 5
```

## 6. 交互设计

### 6.1 基础操作

| 操作 | 行为 |
|---|---|
| 单击节点 | 选中节点，并定位左侧树 |
| 双击节点 | 打开对应 tab |
| 拖拽节点 | 调整局部布局 |
| 拖动画布 | 平移 |
| 滚轮 | 缩放 |
| Esc | 退出聚焦 |
| 搜索 | 高亮匹配节点，并弱化其他节点 |

### 6.2 聚焦行为

单击节点后进入轻聚焦：

- 当前节点高亮。
- 一跳关系保持正常透明度。
- 二跳关系半透明。
- 其他节点低透明。
- 右侧展示影响摘要。

用户可以切换影响方向：

```text
全部关系 / 我依赖的 / 依赖我的
```

MVP 优先实现“全部关系 + 一跳/二跳透明度”即可。

### 6.3 跳转规则

图谱不能替代树和 tab，而是增强它们：

- 单击节点：树图同步展开到目标位置。
- 双击节点：打开 tab。
- 右侧“查看源码”：打开 Header tab 并滚动到声明位置。
- 右侧“编辑字段”：跳到中间表格，不在图谱编辑。

## 7. 数据计算

### 7.1 影响力分数

节点大小不应简单取字段数，而应取综合影响力。

建议公式：

```text
impactScore =
  inboundReferences * 3
  + outboundReferences * 1
  + diagnosticWeight * 2
  + layoutRiskWeight * 2
```

MVP 可以先用：

```text
impactScore = inboundReferences * 3 + outboundReferences
```

### 7.2 风险等级

风险等级用于节点外环。

```text
critical:
  解析错误 / breaking diff / 严重布局异常

warning:
  padding 较高 / lint warning / 缺失关键元数据

normal:
  无明显风险
```

### 7.3 关系方向

边方向统一：

```text
使用者 -> 被引用类型
```

例如：

```cpp
struct RadarTrack {
  Timestamp timestamp;
};
```

图谱边：

```text
RadarTrack -> Timestamp
```

这样在评估修改 `Timestamp` 的影响时，需要反向查找入边。

## 8. MVP 实施范围

当前已有：

- 左侧 rail 入口。
- Canvas 2.5D 动态图谱。
- 单击定位树。
- 双击打开 tab。
- 基础类型引用边。

下一步只建议做 5 件事：

### 8.1 节点影响力编码

- 节点大小改为 impactScore。
- 被引用越多，节点越大。
- 避免所有节点大小接近导致图谱没有信息层级。

### 8.2 聚焦与邻域高亮

- 单击节点后高亮一跳关系。
- 二跳弱显示。
- 无关节点降透明。

这是最能提升“可读性”的功能。

### 8.3 右侧图谱上下文属性

- 图谱视图下右侧展示选中节点的影响摘要。
- 不再只显示普通属性。

这是最能提升“有用性”的功能。

### 8.4 布局风险外环

- 接入 P4 layout 数据。
- 显示 size、align、padding。
- 高 padding 或异常布局节点外环变黄/红。

这是最符合 ProtoVault 差异化价值的功能。

### 8.5 图谱搜索

- 搜索名称、Header、字段引用。
- 匹配节点高亮。
- 非匹配节点不直接删除，而是降低透明度。

这比增加更多模式更有价值。

## 9. 明确延后

以下功能不进入近期版本：

- 真 3D WebGL 图谱。
- 图谱内直接编辑节点或字段。
- 字段级全量节点展开。
- include 全量网络默认展示。
- 图谱截图、导出图片。
- 多人协作关系图。
- AI 自动解释图谱。
- 运行期 MQ / Gateway 拓扑。

## 10. 最终验收标准

关系图谱达到可用，不是看起来炫，而是满足：

1. 打开真实工作区后，用户能在 10 秒内看出核心协议类型。
2. 单击任意类型，能看出它依赖谁、谁依赖它。
3. 修改高影响节点前，用户能看到潜在影响范围。
4. 布局风险和诊断能在图谱上形成明确视觉信号。
5. 双击节点能稳定打开 tab。
6. 单击节点能稳定定位左侧树。
7. 图谱不承担编辑，只负责理解、定位、影响分析。

## 11. 一句话设计原则

```text
图谱只做三件事：
看懂关系，发现风险，跳到正确的编辑位置。
```

任何不服务这三件事的功能，都不进入 MVP。
