# ProtoVault 开发更新日志

本文档记录 Codex Agent 在本地推进 ProtoVault MVP 的实际交付、验证结果和已知限制。它用于配合 Git 提交历史，让后续开发能从“做到了什么、为什么这么做、下一步是什么”继续推进。

## 2026-07-01

### 工程与运行环境

- 建立 pnpm workspace，包含桌面端、契约包和 C++ 服务骨架。
- 桌面端使用 Electron、React、TypeScript、Vite。
- C++ 服务使用 CMake，当前以健康检查和测试骨架作为 P0/P1 基础。
- Windows 本机验证通过：VS Build Tools、LLVM/Clang、Electron、Vitest、Playwright。

### 工作区扫描与 Header 解析

- 支持打开任意文件夹作为工作区，不再要求必须直接打开 `headers/` 根目录。
- 递归发现 `.h/.hh/.hpp/.hxx` 文件，并保留空目录到前端树模型。
- 扫描时跳过 `.git`、`.protocol`、`node_modules`、`out`、`dist`、`coverage`、`build*` 等噪声目录。
- 使用 Clang JSON AST 提取 namespace、struct、enum、字段、定长数组和源码位置。
- 修复 include 声明归属错误：`includedFrom` 仅代表包含来源，不能作为声明所在文件。因此 `geometry.hpp` 中的 `Vec3`、`CoordinateFrame` 不再错误挂到 `track.hpp` 下。

### 目录记录

- 每次扫描后写入 `.protocol/workspace.json`。
- 记录内容包括工作区名称、根路径、扫描时间、目录、Header、类型、诊断和计数。
- 写入采用临时文件 + 原子替换，避免半写入状态污染工作区。

### UI 与交互

- 左侧导航已调整为 Obsidian 风格的合并树：目录 → Header → struct/enum → 字段/枚举项。
- 支持目录、Header、类型节点独立展开和收起。
- 支持三栏拖拽调整：导航栏、主工作区、属性栏。
- 工作区打开与示例扫描放到底部 dock，协议操作放在顶部小图标区域。
- 主工作区支持 Header 源码预览、struct 字段表、enum 值表、属性面板和诊断面板。

### Obsidian/Tokyo Night 主题兼容

- 添加 Obsidian CSS variables 兼容层。
- 添加 `user-obsidian-theme.css` 作为本地主题接入点。
- 当前接入本机主题：

```text
D:\my-documents\.obsidian\themes\Tokyo Night\theme.css
```

- ProtoVault 会优先使用 Tokyo Night 的 `--bg`、`--fg`、`--cyan`、`--blue`、`--comment` 等变量，并映射到通用 Obsidian token，例如 `--background-primary`、`--text-normal`、`--interactive-accent`。

### 基础结构化编辑

- 顶部小图标已接入第一版真实写入动作：
  - 新建 Header。
  - 在当前 Header 创建 `struct`。
  - 给选中的 `struct` 追加字段。
- 写入后自动重新扫描工作区并刷新树、源码预览、类型详情和 `.protocol/workspace.json`。
- 写入路径限制在当前 workspace 内。
- Header/record 写入使用原子替换。

### 验证记录

最近一次完整验证：

```powershell
pnpm typecheck
pnpm test
pnpm build
pnpm test:e2e
```

结果均通过。

### 当前已知限制

- 结构化编辑 UI 已从 `prompt` 迁移到应用内表单；后续应增加预览和撤销。
- `addField` 当前支持简单 `struct Name { ... };` 形式，不处理复杂宏、继承、模板或带特殊格式的声明块。
- Header 生成仍是最小可用版本，尚未实现完整确定性代码生成器、格式化策略和冲突面板。
- Obsidian 主题兼容以 CSS variables 为主，不承诺完整复用 Obsidian DOM 选择器效果。

### 下一步建议

1. 给写入动作增加预览和撤销入口。
2. 建立 IR → Header 的确定性生成器，并让结构化编辑走受控生成路径。
3. 添加文件哈希基线和外部修改冲突检测，进入 P7 双向同步风险门。

## 2026-07-01 P6 Tab 工作台与字段改删

### Tab 工作区

- 中间主工作区加入标签页管理。
- 点击 Header 或协议类型时，会打开或切换对应 tab。
- 支持同时保留多个 Header/Struct/Enum tab，并可关闭 tab。
- 切换 tab 会同步恢复当前主视图、选中类型或源码预览。

### 字段级 CRUD 补齐

- 在 struct 字段表中增加“编辑”入口。
- 字段编辑面板支持：
  - 修改字段类型。
  - 修改字段名称。
  - 删除字段。
- 字段 update/delete 走主进程受控写入 API：
  - 先通过 Clang 扫描定位字段源码行。
  - 修改或删除明确的字段声明行。
  - 写入后重新扫描并刷新 tab、树、字段表和 `.protocol/workspace.json`。
- 对于无法定位源码行的字段，拒绝写入并返回错误，避免猜测式修改。

结果均通过。单元测试覆盖字段新增、修改、删除；E2E 覆盖 tab 打开/切换以及字段编辑面板可见性。

## 2026-07-01 P6 Header/Struct CRUD 补齐

### Header 编辑

- Header 源码预览页新增“编辑 Header”入口。
- 支持：
  - 重命名 Header 相对路径。
  - 删除 Header。
- Header 重命名会移动文件到 workspace 内的新路径；不会自动更新其他 Header 的 include 路径，后续应接入引用更新预览。

### Struct 编辑

- Struct 详情页新增“编辑 Struct”入口。
- 支持：
  - 重命名 struct。
  - 删除 struct。
- 当前重命名只修改 struct 声明名，不自动修改其他字段类型引用；后续需要在语义 Diff/引用分析完成后补引用更新。

### 当前 P6 CRUD 覆盖

- Header：创建、重命名、删除。
- Struct：创建、重命名、删除。
- Field：新增、修改、删除。

## 2026-07-01 P6 快捷编辑与右键菜单

### F2 快捷编辑

- 支持使用 F2 编辑当前选中项：
  - 选中 Header 时打开 Header 编辑面板。
  - 选中 Struct 时打开 Struct 编辑面板。
  - 选中 Struct 字段时打开字段编辑面板。
- 对暂不支持编辑的对象，例如 enum，给出状态提示而不执行写入。

### 右键菜单

- 左侧协议树支持右键上下文菜单。
- Header 右键菜单支持：
  - 编辑 Header。
  - 在该 Header 中新增 Struct。
- Struct 右键菜单支持：
  - 编辑 Struct。
  - 添加字段。
- Field 右键菜单支持：
  - 编辑字段。
  - 添加字段。
- 主树空白区域右键支持新建 Header。

结果均通过。E2E 覆盖 F2 打开 Struct/Field 编辑面板，以及右键菜单打开 Header/Struct/Field 相关编辑入口。

## 2026-07-01 P6 编辑范围跟随选择

### 编辑面板目标同步

- 修正结构化编辑面板的目标对象映射。
- 当编辑面板已经打开时，从左侧树继续点击对象会自动刷新编辑范围：
  - 点击 Header：切换为 Header 编辑，并刷新 Header 相对路径。
  - 点击 Struct：切换为 Struct 编辑，并刷新 Struct 名称。
  - 点击 Field：切换为字段编辑，并刷新字段类型与字段名称。
- “新增 Struct”面板打开时，点击不同 Header/Struct 会刷新目标 Header。
- “添加字段”面板打开时，点击不同 Struct 会刷新字段添加目标。
- 点击 enum 时暂不进入编辑，给出提示并关闭当前结构化编辑目标，避免错误映射。

## 2026-07-01 P6/P8 Enum 编辑与注释元数据纵切

### Enum 结构化编辑

- 将 enum 从“只读展示”提升为和 struct 同级的结构化编辑对象。
- `WorkspaceEnumValueView` 增加稳定 id、源码位置和可选备注字段，避免前端用临时字符串标识枚举项。
- 新增本地业务接口：
  - `protocol:create-enum`
  - `protocol:rename-enum`
  - `protocol:delete-enum`
  - `protocol:add-enum-value`
  - `protocol:update-enum-value`
  - `protocol:delete-enum-value`
- UI 支持：
  - 顶部树工具栏新增 Enum。
  - 选中 enum 后 F2 打开 Enum 编辑面板。
  - 选中 enum value 后 F2 打开枚举项编辑面板。
  - 右键菜单支持编辑 Enum、添加枚举项、编辑枚举项。
  - Editor 表格为 enum value 增加编辑按钮、位置列和注释列。

### 注释元数据

- 新增 `.protocol/meta/metadata.json`，用于保存协议对象的语义注释。
- 当前支持为以下对象记录备注：
  - Struct
  - Field
  - Enum
  - Enum Value
- 注释通过稳定 id 关联，Header 重扫后会重新映射到 IR 视图。
- 右侧 Inspector 增加“注释”编辑区，随左侧树或 Editor 选中对象刷新。

### 已知限制

- 注释当前只持久化在 `.protocol/meta/metadata.json`，尚未生成或回写到 C++ Header 注释。
- Struct/Enum 重命名后稳定 id 会变化，旧注释不会自动迁移；后续需要结合语义 Diff/改名识别做迁移。
- Enum 重命名只修改声明名，不自动更新字段类型引用。

### 验证记录

```powershell
pnpm --filter @protovault/desktop typecheck
pnpm --filter @protovault/desktop test
```

结果通过。单元测试覆盖 Enum/Enum Value CRUD，以及 Struct/Field/Enum/Enum Value 注释跨重扫保留。

## 2026-07-01 P6 中间表格字段直接编辑

### 表格内字段操作

- Struct 的中间 Editor 表格新增直接字段操作能力，不再必须依赖左侧树图选中字段。
- Editor 标题区新增“添加字段”按钮，点击后在表格底部插入草稿行。

  - 取消。
  - 删除字段。
- 保留“面板”按钮作为旧结构化编辑面板入口，方便后续承载更复杂的字段属性。

### 交互原则调整

- 左侧树图继续作为导航和定位入口。
- 中间 Editor 表格成为字段增删改的主操作面。
- 所有表格内保存仍调用同一套受控 Header 写入 API，避免前端伪造状态。
