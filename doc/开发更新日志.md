## 2026-07-10 大型真实 Header 工作区兼容与恢复取消修复

### 真实问题复现

- 使用 `D:\code\xsf-math\include` 的 79 个 Header 通过 Electron IPC 实扫。
- 修复前只得到 1 个 enum，60 个 Header 均被 MSVC `type_traits` 的 `use of undeclared identifier 'byte'` 阻断。
- 根因是 ProtoVault 的最小 `cstddef` 兼容头覆盖了 MSVC `cstddef`，但没有提供 C++17 `std::byte`；MSVC STL 已声明 `__cpp_lib_byte`，形成了不完整的混合标准库环境。

### 修复

- 最小 `cstddef` 兼容头补充 `std::byte`，继续保持目标机器不依赖外部 LLVM/MSVC 安装。
- Header 解析缓存加入 parser cache version；解析环境或恢复算法变化后不再错误复用旧 IR。
- SQLite Header cache schema 升级为 v2，随 IR 一起持久化该 Header 的恢复诊断；warm reopen 不会因为命中缓存而把依赖警告隐藏掉。
- `workspace:restore-last` 将 StrictMode 重复恢复产生的预期 `AbortError` 作为取消返回 `null`，不再污染主进程错误输出。
- 新增非自包含 Header 的受控依赖恢复：仅当未知类型在工作区内存在唯一声明 Header 时，使用临时 `-include` 完成只读 AST 解析，并生成“建议补直接 `#include`”警告；不修改源 Header，不对歧义类型自动猜测。

### `xsf-math` 验证结果

- 79 Headers。
- 235/235 个唯一 struct/enum 定义被识别：216 struct、19 enum、797 fields。
- 首次完整冷扫描约 52 秒；随后跨进程 warm reopen 约 0.3 秒，79 个 Header 全部命中 SQLite cache、0 个重新解析。
- 剩余两条警告均指向同一事实：`xsf_math/core/rcs.hpp` 使用 `vec3` 但未直接包含 `vec3.hpp`；`rcs.hpp` 和依赖它的 `simple_geometries.hpp` 均通过受控恢复完成解析。

### 自动化覆盖

- 新增 MSVC `type_traits` + `std::byte` 兼容测试。
- 新增唯一工作区类型声明的依赖恢复测试。
- 恢复上次工作区 E2E 增加主进程 stderr 断言，确保不再输出 `AbortError`。

## 2026-07-10 `v0.2.0` P18 核心硬化与独立分发

### 持久增量索引

- 新增 `main/workspace-index.ts`，使用 Electron/Node 内置 `node:sqlite`，无需额外 native npm 模块。
- `.protocol/cache/workspace-index.sqlite` 保存 Header 内容 hash、依赖闭包 fingerprint、有效 IR、最后有效 IR 和稳定身份；cache 目录默认不进入 Git。
- 应用重启后的 warm scan 可直接复用 IR；内部 include 变化会使依赖方缓存一起失效。
- 工作区底栏和 Inspector 展示索引引擎、命中数、新解析数、缓存 Header 数和活动身份数。

### 稳定身份与语义 Diff

- Struct、Enum、Field 和枚举值通过精确名称、结构签名与序位回退重关联稳定 ID。
- Header、类型、字段和枚举值改名后不再默认退化为删除加新增。
- Semantic Diff 新增 `type-renamed`、`field-renamed` 和 `enum-value-renamed` 事件。

### 扫描任务与冲突保护

- 同一窗口发起新扫描会取消旧扫描；窗口销毁或写操作开始前也会终止活动扫描。
- 扫描发现、读取、解析和元数据阶段均检查 AbortSignal；Clang 子进程会在取消时被终止。
- 解析失败后可从 SQLite 恢复最后有效 IR，仍允许进入 Header 源码修复。
- 外部变更面板区分“导入磁盘并保留草稿”和“舍弃草稿并导入”；保存继续通过 expected hash 防止静默覆盖。

### 前端边界与版本

- AI Assistant 从 `main.tsx` 拆为 lazy-loaded chunk；主 renderer bundle 从约 952 KB 降到约 929 KB，Assistant chunk 约 28 KB。
- 工作区契约新增索引统计；版本统一升级为 `0.2.0`。
- `v0.2.0` 分发包继续内置 Electron、Git for Windows、Clang 和 MVP 标准头兼容层，目标 Windows 机器无需安装开发环境。

## 2026-07-07 发布包工具链内置：Git 与 Clang 不再要求目标机器预装

### 背景

- 用户要求 Clang 和 Git 相关能力集成到软件本体，安装后的目标计算机不应再额外安装 Git 或 LLVM/Clang。
- 现状中 Header 扫描、保存前语法检查依赖 `clang++.exe`；源代码管理视图依赖 `git.exe`。

### 实现

- 新增 `scripts/prepare-bundled-tools.ps1`：
  - 构建发布包前从构建机采集 LLVM/Clang 与 Git for Windows。
  - 生成 `apps/desktop/vendor-tools/` 构建缓存目录。
  - 该目录被 `.gitignore` 忽略，不进入源码仓库。
- Electron Builder 新增 `extraResources`：
  - `vendor-tools` 打包为 `resources/protovault-tools`。
  - `resources/clang-compat` 打包为 `resources/clang-compat`。
- 运行时工具发现策略调整：
  - Clang 优先使用 `resources/protovault-tools/llvm/bin/clang++.exe`。
  - Git 优先使用 `resources/protovault-tools/git/cmd/git.exe`。
  - 开发模式仍允许通过环境变量或系统安装回退。
- 新增最小 Clang 兼容头：
  - `cstdint`
  - `cstddef`
  - `cstdbool`
  - 用于支持 MVP 范围内的定宽整数和基础类型，避免目标机缺少 MSVC 标准库时连 `<cstdint>` 都无法解析。

- 使用随包 Clang 加 `-nostdinc++` 与 `resources/clang-compat` 检查示例 Header 通过。
- `apps/desktop/release/win-unpacked/resources/protovault-tools/git/cmd/git.exe --version` 可运行。

### 已知限制

- 当前采用完整 Git for Windows 目录和最小 LLVM/Clang 目录，`vendor-tools` 约 701 MB，压缩后的安装包约 298 MB。
- Clang 兼容头只覆盖 MVP 协议解析需要的基础 C++ 头，不等同于完整 C++ 标准库。
- 后续可继续瘦身 Git 目录，或将 Git 能力逐步迁移到纯 JS Git 引擎以进一步减小体积。

## 2026-07-07 发布包修复：主进程不再加载 node_modules 下的 TypeScript 源码

### 背景

- 打开 `release/win-unpacked/ProtoVault.exe` 时，Electron 主进程弹出 JavaScript 错误：
  - `ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`
  - 原因是打包后运行时从 `node_modules/@protovault/contracts` 加载了 `src/index.ts`。
- Electron / Node 不支持对 `node_modules` 内的 TypeScript 文件做 type stripping，因此发布包无法正常启动。

### 修复

- `@protovault/contracts` 的 package exports 从运行时指向 `./src/index.ts` 改为 `./dist/index.js`。
- `@protovault/desktop` 的 `build` 脚本在 `electron-vite build` 前先执行：

```powershell
pnpm --filter @protovault/contracts build
```

- 这样发布包内主进程加载的是 contracts 的 JS 产物，而不是 TS 源码。

### 验证记录

```powershell

pnpm --filter @protovault/desktop dist
```

结果：

- 启动 `apps/desktop/release/win-unpacked/ProtoVault.exe` 成功：

## 2026-07-03 字段注释格式与基础初始化值

### 字段注释格式配置

- 字段注释写回 Header 时改为字段声明同行注释：
  - `std::uint32_t id; // 业务侧稳定 ID`
- Struct / Enum / Enum Value 注释继续保持原有前置注释风格：
  - `/// @brief ...`
- 扫描时兼容读取旧的字段前置注释，但保存字段注释会迁移为同行 `//` 形式。
- 删除字段注释时会移除同行 `//`，并清理紧邻字段上方的旧注释块。

### 字段初始化值

- `WorkspaceFieldView` 增加 `initializer`。
- Clang 扫描后会从字段声明行提取基础初始化值，例如：
  - `std::uint32_t id = 0;`

## 2026-07-06 真实工程 Header 扫描稳定性修复

### 问题

打开 `D:\code\josim\include` 这类真实 C++ include 工作区时，Clang AST JSON 会把 rapidjson、pugixml 等第三方 Header 和深层 include 展开到 stdout。旧实现使用 `execFile` 收集 stdout，即使把 `maxBuffer` 提高到 64 MB，遇到大型 Header 仍可能触发：

```text
stdout maxBuffer length exceeded
```

这不是 Header 本身一定语法错误，而是扫描器把大型依赖 AST 当成一个 stdout 字符串整体读入，导致 Node 子进程缓冲区爆掉。

### 修复

- Clang AST 执行从 `execFile` 改为 `spawn`，stdout 流式写入临时 JSON 文件，避免 Node stdout buffer 直接爆掉。
- 增加 AST 输出大小保护，默认 `256 MB`，可通过 `PROTOVAULT_AST_DUMP_MAX_MB` 调整。
- 将扫描策略改为先从 Header 文本提取 `struct` / `enum` 候选名，再用 `-ast-dump-filter=<TypeName>` 过滤 AST 输出，显著减少第三方 include 噪声。
- 支持解析 Clang filtered dump 产生的多个连续 JSON 对象。
- filtered dump 不保留外层 `NamespaceDecl`，因此根据源码行号恢复 namespace，保证 `demo::radar::RadarTrack` 这类稳定 ID 不退化。
- 默认跳过常见第三方/vendor Header 目录，例如 `rapidjson`、`pugixml`、`boost`、`nlohmann`、`third_party`、`vendor` 等；如确实需要扫描这些目录，可设置 `PROTOVAULT_INCLUDE_VENDOR_HEADERS=true`。
- 默认 Clang 扫描并发从 4 收敛到 2，降低真实大工程下的内存和 IO 峰值。


## 2026-07-10 复杂 Header 解析发布门与错误降级

### 核心解析改造

- Clang 返回非零退出码但已经输出有效 AST 时，不再整 Header 丢弃；解析器会提取健康声明，并过滤诊断范围内的损坏声明。
- 同一 Header 局部损坏时，健康声明采用本次扫描结果，错误声明合并上次有效 IR；无历史基线时仍保留健康声明、源码文件与精确错误位置。
- 部分失败结果不写入“最后有效”缓存，避免把残缺 IR 固化为新基线；修复源码后重新扫描会恢复完整结果。
- 编译诊断从“只取第一条”升级为多条去重收集，支持 `fatal error`、`error`、`warning` 与文件/行/列定位。
- `using` / `typedef` 字段保留源码类型，同时记录 Clang `desugaredQualType`，用于定宽别名布局回退。
- 支持匿名 `typedef struct { ... } Name;`、嵌套 struct 限定名、枚举未显式赋值时的 C++ 递增语义。
- 位域字段保留在 IR 中并标记为布局不支持，避免输出虚假的 size/offset 结论。
- 模板、继承、class、union、宏生成声明、复杂条件编译、成员函数和位域均产生带源码位置的边界诊断；成员函数与 class 按 Header 汇总，避免问题面板被重复信息淹没。

### 新增自动化场景矩阵

新增 `workspace.parser-matrix.test.ts`，覆盖：

1. namespace、定宽类型、`using`、`typedef`、匿名 typedef struct。
2. 单维/多维定长数组、嵌套声明、跨 Header 组合类型引用、字段初始化值。
3. `enum class` 底层类型、显式表达式和隐式递增枚举值。
4. `#pragma pack(push/pop)` 与别名字段布局。
5. 模板、继承、class、union、成员函数、宏、条件编译和位域边界诊断。
6. 同一 Header 中“健康 struct + 损坏 enum”的部分恢复与修复闭环。
7. 损坏兄弟 Header 存在时继续扫描、Lint、生成文档、编辑源码和重新扫描。
8. 唯一缺失依赖恢复、重名依赖拒绝猜测、合法循环 include。
9. 24 个有效 Header 加 1 个损坏 Header 的混合压力扫描、类型计数与单调进度。
10. UTF-8 BOM、CRLF、中文/空格路径、多行块注释和同行字段注释。
