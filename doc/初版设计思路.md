# ProtoVault 数据协议工作台设计说明

## 1. 项目定位

ProtoVault 是一个面向分布式仿真、多系统集成和复杂设备软件项目的数据协议管理工作台。

它以 C++ header 头文件工作卷为入口，提供协议扫描、可视化建模、字段属性管理、协议字节布局分析、代码生成、版本管理、依赖影响分析和本地 AI 辅助能力。工具目标不是单纯替代代码编辑器，而是将项目中的数据协议从“散落在代码中的结构体定义”提升为“可管理、可度量、可追踪、可验证、可演进的工程资产”。

该工具的界面风格参考 Obsidian 的工作卷模式。用户打开的不是单个文件，而是一个协议工作区。工作区直接映射项目中的 header 文件夹，并在此基础上建立协议结构、模块关系、消息流、字段语义和版本演进模型。

## 2. 背景痛点

在当前多系统集成和分布式软件项目中，数据协议通常分散定义在不同模块、不同分系统、不同设备相关的头文件中。协议规模、字段含义、字节大小、通信频率、模块依赖和版本演进缺少统一管理，导致不同角色在项目推进中遇到不同问题。

### 2.1 项目管理人员和设计人员的问题

项目管理人员和设计人员往往无法量化理解项目中的数据协议内容、协议规模、设备关系和分布式软件边界。协议在系统中到底由谁产生、谁消费、数据量有多大、变更影响哪些模块，这些问题通常依赖开发人员口头说明或临时整理。

这种信息不透明会造成决策和沟通问题。例如，某个接口字段变更是否影响多个分系统，某类高频数据是否会造成通信压力，某个设备接入是否需要新增协议或复用已有协议，项目管理人员很难在工具层面直接获得答案。

### 2.2 开发人员的问题

开发人员在调试项目代码时，经常遇到性能问题、功能问题、协议错位、字段解析异常、模块边界不清晰、依赖关系不明确等情况。

由于协议定义分散在多个 header 文件中，开发人员很难快速判断某个结构体被哪些消息使用，某个字段修改会影响哪些模块，某个网络包的字节偏移是否正确，某个结构体中的 padding 是否导致跨平台解析问题。

尤其在二进制协议、共享内存、网络传输、归档回放等场景中，协议的内存布局、序列化布局和真实运行数据可能不一致，导致问题定位困难。

### 2.3 维护和二次开发人员的问题

维护人员和二次开发人员通常需要接手已有协议。由于缺少可视化编辑、字段描述、版本记录和变更影响分析，使用体验较差。

常见问题包括：

协议版本不清楚。
字段含义依赖经验理解。
旧协议和新协议之间缺少语义级差异说明。
修改字段后不知道会影响哪些模块。
生成代码、文档和测试样例需要手工维护。
历史归档数据与当前协议之间缺少映射关系。

因此，工具需要同时服务项目管理、开发调试、维护二开三个场景，而不是只解决 header 文件编辑问题。

## 3. 设计目标

ProtoVault 的目标是构建一个围绕数据协议资产的统一工作台。

第一，支持 Obsidian 类似的工作卷模式。用户可以直接打开一个协议工作区，工作区映射项目中的 header 头文件目录，适用于多系统集成项目对不同分系统、不同模块、不同设备相关协议文件的统一管理。

第二，支持现有 C++ header 文件的反向扫描。工具能够扫描已有头文件，识别结构体、枚举、字段、数组、嵌套结构、include 关系和基础依赖关系，并将其映射为前端可展示、可编辑、可分析的协议模型。

第三，支持前端可视化编辑与代码生成。用户可以通过界面编辑协议结构、字段类型、字段描述、单位、默认值、数组长度、版本状态等信息，并由工具生成受控格式的 C++ header 代码。

第四，支持数据协议属性展示。每个协议结构应展示类似文件属性的信息，包括创建时间、修改时间、字段数量、数据字节大小、字段偏移、对齐填充、引用关系、版本状态和质量检查结果。

第五，支持协议映射与自动生成。工具应支持协议到代码、协议到文档、协议到测试样例、协议到序列化规则的自动生成，减少人工维护成本。

第六，支持本地 AI 辅助。通过本地 AI 模型，对数据源文件、字段命名、协议编辑内容进行自然语言描述推荐，并结合文件属性、协议大小、字段布局和通信频率，提示可能存在的性能瓶颈和优化建议。

第七，支持协议版本管理和语义级差异分析。在 Git 文件版本基础上，进一步提供协议对象级 diff，识别字段新增、删除、改名、类型变化、数组长度变化、枚举值变化、字节大小变化和兼容性风险。

第八，支持依赖影响分析。协议字段发生变化后，工具应能分析受影响的结构体、消息类型、发送模块、消费模块、归档文件、测试用例和生成代码。

## 4. 核心设计理念

ProtoVault 的核心不是“编辑 header 文件”，而是“管理协议资产”。

因此，工具不应让前端直接以 C++ 文本作为唯一数据源。更合理的方式是建立统一的协议 IR 中间模型。

现有 header 文件可以被扫描并导入为协议 IR。前端编辑的是协议 IR。代码生成、文档生成、质量检查、版本 diff、AI 描述和影响分析都基于协议 IR 完成。最终再由协议 IR 生成 C++ header、Markdown 文档、JSON 示例、测试样例和其他目标产物。

这种设计可以避免工具被 C++ 语法细节完全绑定，也为后续支持其他协议格式预留空间。例如未来可以扩展到 Protobuf、FlatBuffers、JSON Schema、自定义二进制协议或前后端接口协议。

## 5. 总体架构设计

ProtoVault 采用前后端分离架构。前端负责交互展示、协议编辑、关系图谱、问题面板和工作区管理。后端负责 header 扫描、协议解析、代码生成、Git 操作、AI 调用、质量检查和本地数据存储。

桌面端建议采用 Electron 进行部署。Electron 可以同时支持桌面端使用和 Web 技术栈渲染，便于实现类似 Obsidian 的工作区界面。前端可以使用 TypeScript、React 或 Vue 实现。编辑器部分可以采用 Monaco Editor 或 CodeMirror。关系图可以采用 React Flow、Cytoscape.js 或 AntV G6。

后端可以采用 C++ 服务、Node.js 服务或混合架构。C++ 后端更适合复用现有工程能力，并可以接入 libclang 完成 C++ header 解析。后续如果需要 Web 化，可以通过 Crow 等 C++ HTTP 框架将后端能力发布为接口，前端通过相同 API 访问。

推荐架构如下：

前端界面层：Electron、TypeScript、React 或 Vue。
编辑器层：Monaco Editor 或 CodeMirror。
图谱展示层：React Flow、Cytoscape.js 或 AntV G6。
本地服务层：C++ 后端服务或 Node.js 主进程服务。
协议解析层：Clang AST 或 libclang。
数据存储层：SQLite 加工作区元数据文件。
版本管理层：Git。
AI 辅助层：本地 Ollama、llama.cpp 或其他本地模型服务。

## 6. 工作卷模式设计

ProtoVault 的工作区可以称为 Protocol Workspace。它对应一个本地目录，内部包含项目已有的 header 文件、协议元数据、生成产物和版本记录。

建议目录结构如下：

```text
project-workspace/
  headers/
    subsystem_a/
    subsystem_b/
    common/
  .protocol/
    ir/
    meta/
    generated/
    reports/
    cache/
  .git/
```

其中，headers 存放项目原始头文件。.protocol/ir 存放结构化协议模型。.protocol/meta 存放字段描述、单位、模块映射、消息映射等补充元数据。.protocol/generated 存放生成的代码和文档。.protocol/reports 存放版本变更报告、质量检查报告和影响分析报告。

这种模式的好处是兼容历史项目。老项目可以先扫描现有 header，再逐步补充协议元数据。新项目可以直接从协议 IR 开始建模，再生成 header 文件。

## 7. 核心功能模块

### 7.1 工作区管理

用户可以创建、打开、扫描和维护协议工作区。

工作区直接映射 header 头文件夹。用户打开工作区后，左侧显示文件树、模块树、协议树和最近修改列表。工具支持扫描已有 header 文件夹，并将扫描结果展示为结构体、枚举、字段和依赖关系。

工作区管理应支持：

打开本地协议目录。
扫描 header 文件夹。
建立协议索引。
展示文件树和协议树。
记录工作区配置。
检测 header 和协议元数据是否同步。
支持重新扫描和增量扫描。

### 7.2 C++ header 反向扫描

工具需要支持对现有 C++ header 文件进行反向映射。扫描目标包括 struct、enum、基础类型字段、定长数组、嵌套结构体、typedef、using、include 关系和部分 pragma pack 信息。

由于 C++ header 复杂度很高，第一版应明确支持范围，避免把工具做成万能 C++ 解析器。

第一版建议支持：

struct。
enum 和 enum class。
基础整数类型。
float 和 double。
定长数组。
嵌套结构体。
有限的 typedef 和 using。
有限的 #pragma pack。
include 依赖关系。

第一版暂不建议完整支持：

模板协议结构。
复杂宏生成结构体。
继承式协议结构。
std::vector、std::string、std::map 作为裸协议字段。
函数、方法和复杂业务逻辑解析。

技术上建议使用 Clang AST 或 libclang，而不是使用正则表达式作为核心解析方案。正则只能用于辅助处理注释和简单文本信息。

### 7.3 协议可视化编辑

前端提供表格化和表单化的协议编辑方式。用户不需要直接手写 C++，而是编辑协议模型。

每个结构体应支持编辑：

结构体名称。
命名空间。
所属模块。
描述信息。
版本状态。
字段列表。
字段顺序。
字段类型。
数组长度。
默认值。
单位。
取值范围。
坐标系。
时间基准。
是否归档。
是否网络发送。
是否参与回放。
是否废弃。

高级用户可以切换到代码视图查看生成后的 header，但应避免前端文本编辑和结构化编辑同时作为主数据源，否则容易产生双向同步问题。

### 7.4 协议属性展示

每个协议结构应展示类似文件属性的信息，但内容应面向协议工程场景。

建议展示：

创建时间。
修改时间。
所属文件。
所属模块。
字段数量。
基础字段数量。
嵌套字段数量。
数组字段数量。
结构体总字节大小。
字段偏移。
字段长度。
padding 字节数。
内存布局大小。
序列化大小。
引用者数量。
被引用结构。
版本状态。
质量检查状态。
AI 建议状态。

其中，数据字节大小需要区分内存布局大小和序列化大小。C++ 的 sizeof 结果可能包含 padding，而网络传输或归档时的实际序列化大小可能不同。工具必须明确展示这两个概念，避免误导用户。

### 7.5 二进制布局分析

二进制布局分析是开发调试的重要功能。工具应展示协议结构体中每个字段的偏移量、字节长度、数组展开大小、嵌套结构体展开结果、对齐填充和总大小。

建议提供字节条视图。用户可以直观看到：

某个字段从第几个字节开始。
字段占用多少字节。
字段后是否存在 padding。
嵌套结构体展开后占用多少空间。
数组字段是否导致结构体过大。
字段修改后后续字段偏移是否变化。

该功能可以帮助开发人员定位网络包错位、二进制归档读取异常、跨平台解析失败、共享内存布局不一致等问题。

### 7.6 字段语义元数据管理

协议字段不应只有名称和类型，还应维护语义元数据。字段语义元数据可以帮助项目管理人员、设计人员和维护人员理解字段业务含义。

建议字段元数据包括：

中文名。
英文名。
自然语言描述。
单位。
取值范围。
默认值。
是否必填。
坐标系。
时间基准。
精度。
刷新频率。
来源模块。
消费模块。
安全等级。
是否归档。
是否参与网络发送。
是否参与回放。
是否参与 UI 展示。
是否允许废弃。

例如，velocity 字段不应只显示为 double[3]。它还应说明单位是 m/s，坐标系是 ENU，含义是目标在站心东北天坐标系下的速度向量，来源模块是 Tracker，消费模块是 Display 和 Recorder，刷新频率是 50 Hz。

### 7.7 协议依赖图

工具应支持结构体、枚举、消息、模块之间的依赖图展示。

依赖图包括：

结构体包含关系。
枚举引用关系。
header include 关系。
结构体被哪些消息使用。
消息由哪些模块发送。
消息由哪些模块消费。
模块之间通过哪些协议通信。
协议变更会影响哪些模块。

第一版可以先做结构体依赖图和 include 关系图。后续再增加消息流图、模块关系图和设备映射图。

### 7.8 消息流图和通信视角

即使通信信道指标暂时不作为第一版核心字段，也建议保留轻量消息流模型。尤其对于 Gateway 和 MQ 两类通信模式，工具可以将协议结构体与消息类型、发送模块、接收模块、传输模式、频率和数据量估算关联起来。

建议消息流模型包括：

消息名称。
消息 ID。
绑定协议结构体。
发送模块。
接收模块。
传输模式。
发送频率。
单包大小。
理论数据吞吐。
是否归档。
是否回放。

例如，RadarRawFrame 由 RadarSim 通过 MQ 发布，Display 和 Recorder 订阅，频率为 20 Hz，单包大小为 128 KB，理论裸数据量为 2.56 MB/s。

这类视图可以帮助项目管理人员理解系统数据压力，也可以帮助开发人员定位高频大包和性能瓶颈。

### 7.9 协议质量检查

工具应提供协议 lint 规则检查。规则分为错误、警告和建议三类，避免规则过严导致工具难以推广。

建议检查内容包括：

禁止协议结构体中直接使用指针。
限制 std::string、std::vector、std::map 等运行期容器进入裸二进制协议。
检查 bool、enum、long、size_t 等跨平台不稳定类型。
检查字段单位是否缺失。
检查坐标系是否缺失。
检查时间基准是否缺失。
检查字段描述是否为空。
检查数组字段是否过大。
检查 padding 是否过多。
检查结构体是否缺少版本号。
检查是否存在重复字段语义。
检查枚举是否缺少 Unknown 或 Invalid 值。
检查大小端策略是否明确。
检查归档字段是否缺少兼容性说明。

质量检查的价值在于提前发现设计阶段的问题，而不是等到代码调试或现场维护阶段才暴露。

### 7.10 语义级版本管理

工具需要基于 Git 做版本管理，但不能只停留在文件 diff 层面。

普通 Git diff 只能显示代码行变化，而协议管理工具应该识别协议对象变化。

语义级 diff 应支持识别：

新增字段。
删除字段。
字段改名。
字段类型变化。
数组长度变化。
枚举值新增。
枚举值重排。
结构体大小变化。
字段偏移变化。
默认值变化。
单位变化。
描述变化。
字段废弃。
兼容性等级变化。

每次协议变更应生成变更报告。报告说明本次变更影响哪些结构体、哪些消息、哪些模块、哪些归档文件、哪些测试样例，以及变更是否兼容。

兼容性可以分为：

兼容变更。
可能不兼容变更。
明确不兼容变更。

例如，字段 targetCount 从 uint16 改为 uint32 后，字段大小增加 2 字节，后续字段偏移可能变化。如果系统采用二进制直接映射读取，那么这是明确不兼容变更。如果系统采用字段级序列化，兼容性则取决于序列化规则和消费者版本。

### 7.11 代码生成和文档生成

工具应基于协议 IR 生成多类产物，而不只是生成 C++ header。

建议支持生成：

C++ header。
序列化代码。
反序列化代码。
默认值初始化代码。
字段反射表。
Markdown 协议文档。
JSON 示例数据。
YAML 配置模板。
测试样例。
二进制样例包。
版本迁移代码。

其中字段反射表非常关键。C++ 原生反射能力不足，如果希望后续支持调试工具、字段级展示、二进制解析和运行期观测，最好在代码生成时同步生成字段元信息。

字段元信息至少包括：

结构体名。
字段名。
字段类型。
字段偏移。
字段大小。
数组长度。
单位。
描述。
版本状态。

### 7.12 调试辅助工具

为了服务开发调试，工具应提供协议数据调试能力。

建议支持：

生成默认样例数据。
生成边界样例数据。
生成最大长度样例数据。
查看二进制 dump。
解析二进制数据包。
按字段展示原始字节。
按字段展示解析结果。
检查字段偏移是否异常。
识别历史归档文件对应的协议版本。
对比新旧版本协议对同一数据包的解析差异。

该功能可以帮助开发人员快速定位网络数据乱码、字段错位、结构体大小不一致、归档回放失败等问题。

### 7.13 数据量估算和性能提示

工具需要结合协议大小、消息频率、发送模块、接收模块和传输方式，估算协议在系统中的数据压力。

建议展示：

单协议大小。
单消息大小。
单模块发送数据量。
单模块接收数据量。
单设备数据量。
单链路数据量。
高频协议列表。
大包协议列表。
高风险协议列表。

例如，某协议单包 4 KB，如果每秒发送 2000 次，则理论裸数据量约为 8 MB/s。再结合订阅者数量和序列化开销，工具可以提示它可能是性能瓶颈。

性能建议可以包括：

拆分冷热字段。
降低发送频率。
改为增量发送。
压缩大数组字段。
避免无意义 padding。
减少重复字段。
将调试字段从正式链路中移除。
将低频字段改为按需请求。

### 7.14 本地 AI 辅助

ProtoVault 可以接入本地 AI 模型，提供辅助能力。但 AI 不应成为协议定义的权威来源。

建议 AI 功能包括：

字段描述推荐。
协议摘要生成。
字段命名建议。
单位和坐标系缺失提示。
协议变更总结。
性能风险解释。
文档草稿生成。
自然语言问答。

例如，用户可以问：

哪个协议最大？
哪些协议使用了 ENU 坐标系？
某个字段被哪些模块使用？
最近一次协议变更影响了哪些消费者？
某个高频消息为什么可能造成性能问题？

AI 输出应标记为推荐内容，需要人工确认。对于字段含义、兼容性判断、协议修改和代码生成，最终结果应由规则引擎、编译校验和人工评审确认。

评审者。

## 10. 关键风险和边界

### 10.1 不要把第一版做成万能 C++ 解析器

C++ header 复杂度很高。如果第一版试图支持所有宏、模板、继承和条件编译，项目复杂度会迅速失控。

应明确支持协议 C++ 子集，并对不支持的结构给出提示。

### 10.2 不要让 header 文本成为唯一主数据源

如果前端直接编辑 header，后端又反向解析 header，容易出现双向同步问题。

更稳妥的方式是：

已有 header 导入为协议 IR。
前端编辑协议 IR。
工具从协议 IR 生成 header。

### 10.3 不要只做文件级 Git 版本管理

维护人员真正关心的是协议语义变化，而不是某几行代码变化。工具必须提供字段级、结构体级、消息级的语义 diff。

### 10.4 不要让 AI 替代规则校验

AI 可以生成描述、解释风险和总结变更，但不能直接决定协议是否正确，也不能作为兼容性判断的唯一依据。

### 10.5 必须区分内存布局和序列化布局

C++ sizeof 结果和网络发送大小不一定相同。工具必须区分内存布局大小、序列化大小和实际运行数据量。

### 10.6 必须结合频率理解性能风险

协议大小本身不能直接等价于性能压力。一个大协议低频发送可能没有问题，一个小协议超高频发送可能成为瓶颈。数据量估算必须结合发送频率、订阅者数量和传输方式。

## 11. 项目价值

对于项目管理人员，ProtoVault 可以把协议规模、模块边界、数据流向和变更影响量化展示，减少沟通依赖和决策盲区。

对于设计人员，ProtoVault 可以帮助建立清晰的数据契约，明确字段语义、单位、坐标系、时间基准和协议边界。

对于开发人员，ProtoVault 可以帮助定位协议布局、字段偏移、依赖关系、性能瓶颈和代码生成问题。

对于维护和二次开发人员，ProtoVault 可以提供协议版本、变更记录、影响分析、可视化编辑和文档生成能力，降低接手成本。

## 12. 总结定位

ProtoVault 不是一个简单的 header 编辑器，而是一个面向分布式软件项目的数据协议资产管理工作台。

它以 Obsidian 类似的工作卷界面为入口，以 C++ header 扫描和协议 IR 建模为核心，以可视化编辑、字节布局分析、版本治理、代码生成、依赖影响分析和本地 AI 辅助为主要能力，最终帮助团队把数据协议从代码细节变成可管理的工程资产。
