# ProtoVault 数据协议工作台设计说明

## 1. 项目定位

ProtoVault 是一个面向分布式仿真、多系统集成和复杂设备软件项目的数据协议管理工作台。

它以 C++ header 头文件工作卷为入口，提供协议扫描、可视化建模、字段属性管理、协议字节布局分析、代码生成、版本管理、依赖影响分析和本地 AI 辅助能力。工具目标不是单纯替代代码编辑器，而是将项目中的数据协议从“散落在代码中的结构体定义”提升为“可管理、可度量、可追踪、可验证、可演进的工程资产”。

该工具的界面风格参考 Obsidian 的工作卷模式。用户打开的不是单个文件，而是一个协议工作区。工作区直接映射项目中的 header 文件夹，并在此基础上建立协议结构、模块关系、协议网络地图、字段语义和版本演进模型。

随着设计进一步收敛，ProtoVault 不应只把“数据流”作为编辑主对象。对于大型仿真和分布式系统，真实工程事实往往是一张由实体节点、通信链路和协议载荷共同构成的数据链路网络。某条业务数据流只是这张网络的一个观察视角或分析结论。因此，ProtoVault 的长期核心能力应升级为 Protocol Network Map，即协议网络地图：用户主要维护节点、链路和链路上的协议绑定，系统再基于这些事实生成业务数据流视图、影响分析视图和性能瓶颈视图。

## 2. 背景痛点

在当前多系统集成和分布式软件项目中，数据协议通常分散定义在不同模块、不同分系统、不同设备相关的头文件中。协议规模、字段含义、字节大小、通信频率、模块依赖和版本演进缺少统一管理，导致不同角色在项目推进中遇到不同问题。

### 2.1 项目管理人员和设计人员的问题

项目管理人员和设计人员往往无法量化理解项目中的数据协议内容、协议规模、设备关系和分布式软件边界。协议在系统中到底由谁产生、谁消费、数据量有多大、变更影响哪些模块，这些问题通常依赖开发人员口头说明或临时整理。

这种信息不透明会造成决策和沟通问题。例如，某个接口字段变更是否影响多个分系统，某类高频数据是否会造成通信压力，某个设备接入是否需要新增协议或复用已有协议，项目管理人员很难在工具层面直接获得答案。

### 2.2 开发人员的问题

开发人员在调试项目代码时，经常遇到性能问题、功能问题、协议错位、字段解析异常、模块边界不清晰、依赖关系不明确等情况。

由于协议定义分散在多个 header 文件中，开发人员很难快速判断某个结构体被哪些协议绑定使用，某个字段修改会影响哪些节点和链路，某个网络包的字节偏移是否正确，某个结构体中的 padding 是否导致跨平台解析问题。

尤其在二进制协议、共享内存、网络传输、归档回放等场景中，协议的内存布局、序列化布局和真实运行数据可能不一致，导致问题定位困难。

### 2.3 维护和二次开发人员的问题

维护人员和二次开发人员通常需要接手已有协议。由于缺少可视化编辑、字段描述、版本记录和变更影响分析，使用体验较差。

常见问题包括：

协议版本不清楚。
字段含义依赖经验理解。
旧协议和新协议之间缺少语义级差异说明。
修改字段后不知道会影响哪些模块。
生成代码、文档和测试样例需要手工维护。
历史归档数据与当前协议之间缺少映射关系。

因此，工具需要同时服务项目管理、开发调试、维护二开三个场景，而不是只解决 header 文件编辑问题。

### 2.4 大型仿真系统的数据链路问题

在大型仿真、半实物仿真和多节点分布式系统中，项目并不是由几条孤立的数据流组成，而是由大量实体节点和通信链路构成一张数据链路网络。节点可能是仿真主控、传感器模型、目标生成器、环境模型、算法服务、可视化终端、记录回放服务、网关进程或真实硬件设备。节点之间通过 UDP、TCP、DDS、共享内存、文件、MQ 或自定义总线传输多种协议。

如果工具只在协议类型上填写“生产者”和“消费者”，会把协议结构和系统拓扑混在一起。某个协议可能在多条链路上被传输，同一对节点之间也可能同时传输多个协议。一次大型仿真中还会存在目标链路、遥测链路、日志链路、控制链路、回放链路等多个业务视角，如果把这些业务视角都作为主要编辑对象，模型会变得分散且难以维护。

更稳妥的做法是：用户编辑系统网络事实，工具从事实网络中生成数据流理解。也就是说，节点和链路是主数据；协议绑定是链路上的载荷；数据流、性能瓶颈和智能体总结是派生视图。

## 3. 设计目标

ProtoVault 的目标是构建一个围绕数据协议资产的统一工作台。

第一，支持 Obsidian 类似的工作卷模式。用户可以直接打开一个协议工作区，工作区映射项目中的 header 头文件目录，适用于多系统集成项目对不同分系统、不同模块、不同设备相关协议文件的统一管理。

第二，支持现有 C++ header 文件的反向扫描。工具能够扫描已有头文件，识别结构体、枚举、字段、数组、嵌套结构、include 关系和基础依赖关系，并将其映射为前端可展示、可编辑、可分析的协议模型。

第三，支持前端可视化编辑与代码生成。用户可以通过界面编辑协议结构、字段类型、字段描述、单位、默认值、数组长度、版本状态等信息，并由工具生成受控格式的 C++ header 代码。

第四，支持数据协议属性展示。每个协议结构应展示类似文件属性的信息，包括创建时间、修改时间、字段数量、数据字节大小、字段偏移、对齐填充、引用关系、版本状态和质量检查结果。

第五，支持协议映射与自动生成。工具应支持协议到代码、协议到文档、协议到测试样例、协议到序列化规则的自动生成，减少人工维护成本。

第六，支持本地 AI 辅助。通过本地 AI 模型，对数据源文件、字段命名、协议编辑内容进行自然语言描述推荐，并结合文件属性、协议大小、字段布局和通信频率，提示可能存在的性能瓶颈和优化建议。

第七，支持协议版本管理和语义级差异分析。在 Git 文件版本基础上，进一步提供协议对象级 diff，识别字段新增、删除、改名、类型变化、数组长度变化、枚举值变化、字节大小变化和兼容性风险。

第八，支持依赖影响分析。协议字段发生变化后，工具应能分析受影响的结构体、协议文件、协议绑定、网络节点、通信链路、归档文件、测试用例和生成代码。

第九，支持协议网络地图。工具应允许用户维护系统中的实体节点、节点之间的通信链路，以及链路上传输的协议绑定。协议类型不再直接拥有生产者和消费者，而是作为链路载荷被绑定到具体通信关系上。

第十，支持从协议网络事实中生成业务视图。业务数据流、高带宽风险、协议影响范围、关键路径和后续智能体分析结论，都应从节点、链路和协议绑定中派生，而不是作为第一层编辑事实。

## 4. 核心设计理念

ProtoVault 的核心不是“编辑 header 文件”，而是“管理协议资产”。

因此，工具不应让前端直接以 C++ 文本作为唯一数据源。更合理的方式是建立统一的协议 IR 中间模型。

现有 header 文件可以被扫描并导入为协议 IR。前端编辑的是协议 IR。代码生成、文档生成、质量检查、版本 diff、AI 描述和影响分析都基于协议 IR 完成。最终再由协议 IR 生成 C++ header、Markdown 文档、JSON 示例、测试样例和其他目标产物。

这种设计可以避免工具被 C++ 语法细节完全绑定，也为后续支持其他协议格式预留空间。例如未来可以扩展到 Protobuf、FlatBuffers、JSON Schema、自定义二进制协议或前后端接口协议。

在协议 IR 之外，ProtoVault 还需要建立独立的系统网络模型。协议 IR 回答“数据结构长什么样”，网络模型回答“数据在系统中如何流动”。两者通过 Protocol Binding 连接：某条链路可以绑定一个或多个协议类型，并记录 topic、端口、频率、批量大小、延迟预算和估算带宽。

最终模型应遵循以下原则：

```text
协议结构是数据格式事实。
实体节点是系统部署事实。
通信链路是节点连接事实。
协议绑定是链路载荷事实。
业务数据流是从事实网络中提炼出来的视图。
```

因此，生产者和消费者不应直接写在协议类型上。更合理的表达是：某个节点通过某条链路向另一个节点传输某个协议载荷。这样既能表达多个节点之间的真实通信关系，也能支持后续性能估算、瓶颈分析和 AI 总结。

## 5. 总体架构设计

ProtoVault 采用前后端分离架构。前端负责交互展示、协议编辑、关系图谱、问题面板和工作区管理。后端负责 header 扫描、协议解析、代码生成、Git 操作、AI 调用、质量检查和本地数据存储。

桌面端建议采用 Electron 进行部署。Electron 可以同时支持桌面端使用和 Web 技术栈渲染，便于实现类似 Obsidian 的工作区界面。前端可以使用 TypeScript、React 或 Vue 实现。编辑器部分可以采用 Monaco Editor 或 CodeMirror。小规模结构依赖图可以采用 React Flow、Cytoscape.js 或 AntV G6；大规模协议网络地图应优先考虑 Canvas/WebGL 力导向渲染，以保证滚轮缩放、拖拽平移和节点动画足够流畅。

后端可以采用 C++ 服务、Node.js 服务或混合架构。C++ 后端更适合复用现有工程能力，并可以接入 libclang 完成 C++ header 解析。后续如果需要 Web 化，可以通过 Crow 等 C++ HTTP 框架将后端能力发布为接口，前端通过相同 API 访问。

推荐架构如下：

前端界面层：Electron、TypeScript、React 或 Vue。
编辑器层：Monaco Editor 或 CodeMirror。
图谱展示层：小图使用 React Flow、Cytoscape.js 或 AntV G6，大图使用 Canvas/WebGL 力导向渲染。
本地服务层：C++ 后端服务或 Node.js 主进程服务。
协议解析层：Clang AST 或 libclang。
数据存储层：SQLite 加工作区元数据文件。
版本管理层：Git。
AI 辅助层：本地 Ollama、llama.cpp 或其他本地模型服务。

## 6. 工作卷模式设计

ProtoVault 的工作区可以称为 Protocol Workspace。它对应一个本地目录，内部包含项目已有的 header 文件、协议元数据、生成产物和版本记录。

建议目录结构如下：

```text
project-workspace/
  headers/
    subsystem_a/
    subsystem_b/
    common/
  .protocol/
    ir/
    meta/
    network/
    views/
    generated/
    reports/
    cache/
  .git/
```

其中，headers 存放项目原始头文件。.protocol/ir 存放结构化协议模型。.protocol/meta 存放字段描述、单位、取值范围、注释映射等补充元数据。.protocol/network 存放实体节点、通信链路、协议绑定和软硬件画像。.protocol/views 存放用户保存的派生视图配置，例如某条业务数据流、某个关键链路或某个性能风险视角。.protocol/generated 存放生成的代码和文档。.protocol/reports 存放版本变更报告、质量检查报告、影响分析报告和网络性能估算报告。

这种模式的好处是兼容历史项目。老项目可以先扫描现有 header，再逐步补充协议元数据。新项目可以直接从协议 IR 开始建模，再生成 header 文件。

## 7. 核心功能模块

### 7.1 工作区管理

用户可以创建、打开、扫描和维护协议工作区。

工作区直接映射 header 头文件夹。用户打开工作区后，左侧显示文件树、模块树、协议树和最近修改列表。工具支持扫描已有 header 文件夹，并将扫描结果展示为结构体、枚举、字段和依赖关系。

工作区管理应支持：

打开本地协议目录。
扫描 header 文件夹。
建立协议索引。
展示文件树和协议树。
记录工作区配置。
检测 header 和协议元数据是否同步。
支持重新扫描和增量扫描。

### 7.2 C++ header 反向扫描

工具需要支持对现有 C++ header 文件进行反向映射。扫描目标包括 struct、enum、基础类型字段、定长数组、嵌套结构体、typedef、using、include 关系和部分 pragma pack 信息。

由于 C++ header 复杂度很高，第一版应明确支持范围，避免把工具做成万能 C++ 解析器。

第一版建议支持：

struct。
enum 和 enum class。
基础整数类型。
float 和 double。
定长数组。
嵌套结构体。
有限的 typedef 和 using。
有限的 #pragma pack。
include 依赖关系。

第一版暂不建议完整支持：

模板协议结构。
复杂宏生成结构体。
继承式协议结构。
std::vector、std::string、std::map 作为裸协议字段。
函数、方法和复杂业务逻辑解析。

技术上建议使用 Clang AST 或 libclang，而不是使用正则表达式作为核心解析方案。正则只能用于辅助处理注释和简单文本信息。

### 7.3 协议可视化编辑

前端提供表格化和表单化的协议编辑方式。用户不需要直接手写 C++，而是编辑协议模型。

每个结构体应支持编辑：

结构体名称。
命名空间。
所属模块。
描述信息。
版本状态。
字段列表。
字段顺序。
字段类型。
数组长度。
默认值。
基础初始化值。
单位。
取值范围。
坐标系。
时间基准。
是否归档。
是否网络发送。
是否参与回放。
是否废弃。

高级用户可以切换到代码视图查看生成后的 header，但应避免前端文本编辑和结构化编辑同时作为主数据源，否则容易产生双向同步问题。

注释和初始化值必须纳入双向同步规则。字段注释建议写在字段声明同行，采用 `// 描述` 形式，便于源码阅读和稳定回写；结构体、枚举和枚举项可以继续采用前置 `/// @brief ...`。扫描时应兼容读取已有的 `//`、`///` 和多行注释，保存时按工作区配置生成统一风格。基础类型字段应支持初始化值，例如 `std::uint32_t id = 0;`、`bool enabled = false;` 和 `std::uint8_t payload[8] = {};`，并在结构化表格、源码视图和生成 Header 之间保持一致。

### 7.4 协议属性展示

每个协议结构应展示类似文件属性的信息，但内容应面向协议工程场景。

建议展示：

创建时间。
修改时间。
所属文件。
所属模块。
字段数量。
基础字段数量。
嵌套字段数量。
数组字段数量。
结构体总字节大小。
字段偏移。
字段长度。
padding 字节数。
内存布局大小。
序列化大小。
引用者数量。
被引用结构。
版本状态。
质量检查状态。
AI 建议状态。

其中，数据字节大小需要区分内存布局大小和序列化大小。C++ 的 sizeof 结果可能包含 padding，而网络传输或归档时的实际序列化大小可能不同。工具必须明确展示这两个概念，避免误导用户。

### 7.5 二进制布局分析

二进制布局分析是开发调试的重要功能。工具应展示协议结构体中每个字段的偏移量、字节长度、数组展开大小、嵌套结构体展开结果、对齐填充和总大小。

建议提供字节条视图。用户可以直观看到：

某个字段从第几个字节开始。
字段占用多少字节。
字段后是否存在 padding。
嵌套结构体展开后占用多少空间。
数组字段是否导致结构体过大。
字段修改后后续字段偏移是否变化。

该功能可以帮助开发人员定位网络包错位、二进制归档读取异常、跨平台解析失败、共享内存布局不一致等问题。

### 7.6 字段语义元数据管理

协议字段不应只有名称和类型，还应维护语义元数据。字段语义元数据可以帮助项目管理人员、设计人员和维护人员理解字段业务含义。

建议字段元数据包括：

中文名。
英文名。
自然语言描述。
单位。
取值范围。
默认值。
是否必填。
坐标系。
时间基准。
精度。
刷新频率。
安全等级。
是否归档。
是否参与回放。
是否参与 UI 展示。
是否允许废弃。
网络绑定状态。
关联链路数量。

字段元数据只描述字段本身，不直接维护来源模块和消费模块。来源、去向、发送频率、传输方式和订阅关系应由协议网络地图表达，再反向汇总为字段或结构体的只读属性。

例如，velocity 字段不应只显示为 double[3]。它还应说明单位是 m/s，坐标系是 ENU，含义是目标在站心东北天坐标系下的速度向量。如果该字段所属结构体在 RadarModel 到 Tracker 的链路上以 50 Hz 传输，那么这个来源和频率来自协议绑定，而不是字段自身。

### 7.7 协议依赖图

工具应支持协议对象之间的依赖图展示。该图谱回答“协议结构之间如何引用”，不直接承担系统拓扑建模。

依赖图包括：

结构体包含关系。
枚举引用关系。
字段类型引用关系。
header include 关系。
结构体被哪些结构体引用。
协议变更会影响哪些协议文件。

第一版可以先做结构体依赖图、字段引用图和 include 关系图。模块通信、设备映射、数据流向和性能链路应进入协议网络地图，而不是塞进协议依赖图。

### 7.8 协议网络地图和通信链路视角

ProtoVault 应在协议 IR 之外提供协议网络地图。它不是普通的消息流图，而是一套独立的系统事实模型，用于描述实体节点、通信链路和链路上传输的协议载荷。

协议网络地图的第一层可编辑对象包括：

实体节点 NetworkNode。
通信链路 NetworkLink。
协议绑定 ProtocolBinding。
节点软硬件画像 NodeProfile。
派生视图 FlowView。

其中，NetworkNode 表示系统中的真实或逻辑实体，可以是仿真主控、传感器模型、目标生成器、环境模型、算法服务、可视化终端、记录回放服务、网关进程、外部系统或真实硬件设备。

NetworkNode 建议包含：

节点名称。
节点类型。
节点角色。
所属分系统。
主机或部署位置。
进程名或可执行文件。
软件运行时。
线程或队列模型。
CPU、GPU、内存、网络和磁盘画像。
操作系统和容器信息。
备注和责任人。

NetworkLink 表示两个节点之间的通信关系。链路可以是 UDP、TCP、DDS、共享内存、文件、MQ、自定义总线或手工标记的抽象链路。

NetworkLink 建议包含：

源节点。
目标节点。
传输方式。
topic、端口、队列名或文件路径。
QoS 或可靠性策略。
延迟预算。
带宽上限。
是否可回放。
是否归档。
是否关键链路。

ProtocolBinding 表示某条链路上承载的协议类型。一个链路可以绑定多个协议，一个协议也可以被绑定到多条链路。

ProtocolBinding 建议包含：

绑定名称。
链路 ID。
协议类型 ID。
业务数据名。
发送频率。
批量大小。
单包大小。
估算带宽。
峰值倍数。
关键等级。
兼容性要求。
归档策略。
备注。

FlowView 不应作为主数据，而应作为从网络事实中提炼出来的视图。它可以表达某个业务场景下的目标链路、遥测链路、控制链路、日志链路、回放链路或 AI 生成的关键路径总结。

```text
NetworkNode  --NetworkLink-->  NetworkNode
                     |
                     +-- ProtocolBinding --> Struct / Enum / ProtocolFile
```

例如，RadarModel 通过 UDP 向 Tracker 发送 RadarDetectionFrame，频率 50 Hz；同一条链路还可以低频发送 RadarHealthStatus，频率 1 Hz。此时 RadarDetectionFrame 不需要直接写“生产者是 RadarModel、消费者是 Tracker”。更准确的事实是：某条链路绑定了某个协议载荷。

这套模型可以帮助用户回答：

某个协议在哪些链路上传输。
某个节点发送和接收了哪些协议。
某条链路的理论数据量是多少。
某个协议变更会影响哪些节点和链路。
某个硬件节点是否可能成为吞吐或延迟瓶颈。
某个业务数据流到底经过了哪些节点。

对于已有的生产者、消费者配置，迁移策略应是生成默认 NetworkNode 和 NetworkLink，再把原协议映射为 ProtocolBinding。后续用户可以在网络视图中继续细化，而不是继续把拓扑信息写回协议类型。

### 7.9 协议质量检查

工具应提供协议 lint 规则检查。规则分为错误、警告和建议三类，避免规则过严导致工具难以推广。

建议检查内容包括：

禁止协议结构体中直接使用指针。
限制 std::string、std::vector、std::map 等运行期容器进入裸二进制协议。
检查 bool、enum、long、size_t 等跨平台不稳定类型。
检查字段单位是否缺失。
检查坐标系是否缺失。
检查时间基准是否缺失。
检查字段描述是否为空。
检查数组字段是否过大。
检查 padding 是否过多。
检查结构体是否缺少版本号。
检查是否存在重复字段语义。
检查枚举是否缺少 Unknown 或 Invalid 值。
检查大小端策略是否明确。
检查归档字段是否缺少兼容性说明。

质量检查的价值在于提前发现设计阶段的问题，而不是等到代码调试或现场维护阶段才暴露。

### 7.10 语义级版本管理

工具需要基于 Git 做版本管理，但不能只停留在文件 diff 层面。

普通 Git diff 只能显示代码行变化，而协议管理工具应该识别协议对象变化。

语义级 diff 应支持识别：

新增字段。
删除字段。
字段改名。
字段类型变化。
数组长度变化。
枚举值新增。
枚举值重排。
结构体大小变化。
字段偏移变化。
默认值变化。
单位变化。
描述变化。
字段废弃。
兼容性等级变化。

每次协议变更应生成变更报告。报告说明本次变更影响哪些结构体、哪些协议绑定、哪些节点、哪些链路、哪些归档文件、哪些测试样例，以及变更是否兼容。

兼容性可以分为：

兼容变更。
可能不兼容变更。
明确不兼容变更。

例如，字段 targetCount 从 uint16 改为 uint32 后，字段大小增加 2 字节，后续字段偏移可能变化。如果系统采用二进制直接映射读取，那么这是明确不兼容变更。如果系统采用字段级序列化，兼容性则取决于序列化规则和读取端版本。

### 7.11 代码生成和文档生成

工具应基于协议 IR 生成多类产物，而不只是生成 C++ header。

建议支持生成：

C++ header。
序列化代码。
反序列化代码。
默认值初始化代码。
字段反射表。
Markdown 协议文档。
JSON 示例数据。
YAML 配置模板。
测试样例。
二进制样例包。
版本迁移代码。

其中字段反射表非常关键。C++ 原生反射能力不足，如果希望后续支持调试工具、字段级展示、二进制解析和运行期观测，最好在代码生成时同步生成字段元信息。

字段元信息至少包括：

结构体名。
字段名。
字段类型。
字段偏移。
字段大小。
数组长度。
单位。
描述。
版本状态。

### 7.12 调试辅助工具

为了服务开发调试，工具应提供协议数据调试能力。

建议支持：

生成默认样例数据。
生成边界样例数据。
生成最大长度样例数据。
查看二进制 dump。
解析二进制数据包。
按字段展示原始字节。
按字段展示解析结果。
检查字段偏移是否异常。
识别历史归档文件对应的协议版本。
对比新旧版本协议对同一数据包的解析差异。

该功能可以帮助开发人员快速定位网络数据乱码、字段错位、结构体大小不一致、归档回放失败等问题。

### 7.13 数据量估算和性能提示

工具需要结合协议布局、协议绑定、链路属性和节点画像，估算协议在系统中的数据压力。估算不应只停留在“某个结构体有多大”，而应回答“这个结构体在什么链路上，以什么频率，被哪些节点传输”。

建议展示：

单协议大小。
单绑定载荷大小。
单链路裸数据量。
单链路估算数据量。
单节点发送数据量。
单节点接收数据量。
单主机或设备数据量。
单业务视图数据量。
单链路数据量。
高频协议列表。
大包协议列表。
关键链路列表。
高风险协议列表。

估算基础公式可以先采用：

```text
裸数据量 = 协议序列化大小 * 发送频率 * 批量大小
链路估算数据量 = 裸数据量 * 传输开销系数 * 峰值倍数
节点发送数据量 = 该节点所有出向协议绑定的数据量之和
节点接收数据量 = 该节点所有入向协议绑定的数据量之和
```

例如，某协议单包 4 KB，如果在 RadarModel 到 Tracker 的 UDP 链路上每秒发送 2000 次，则理论裸数据量约为 8 MB/s。再结合 UDP/IP 开销、峰值倍数、目标节点处理能力和链路带宽，工具可以提示它可能是吞吐瓶颈或调度风险。

性能建议可以包括：

拆分冷热字段。
降低发送频率。
改为增量发送。
压缩大数组字段。
避免无意义 padding。
减少重复字段。
将调试字段从正式链路中移除。
将低频字段改为按需请求。
将多接收节点广播拆分为分层转发。
将高频链路迁移到共享内存或更合适的传输方式。
将 CPU 密集型节点拆分或调整部署位置。

由于节点画像记录了 CPU、GPU、内存、网络、磁盘、运行时和部署信息，工具还可以逐步从“协议数据量提示”升级为“工程瓶颈提示”。例如同样是 8 MB/s，跑在本机共享内存、跨主机 UDP、嵌入式板卡网口或虚拟化容器中的风险完全不同，必须结合节点和链路上下文判断。

### 7.14 本地 AI 辅助

ProtoVault 可以接入本地 AI 模型，提供辅助能力。但 AI 不应成为协议定义的权威来源。

建议 AI 功能包括：

字段描述推荐。
协议摘要生成。
字段命名建议。
单位和坐标系缺失提示。
协议变更总结。
性能风险解释。
文档草稿生成。
自然语言问答。

例如，用户可以问：

哪个协议最大？
哪些协议使用了 ENU 坐标系？
某个字段被哪些模块使用？
最近一次协议变更影响了哪些节点和链路？
某个高频协议绑定为什么可能造成性能问题？

AI 输出应标记为推荐内容，需要人工确认。对于字段含义、兼容性判断、协议修改和代码生成，最终结果应由规则引擎、编译校验和人工评审确认。

### 7.15 权限和评审流程

如果工具用于正式项目，可以设计轻量权限和评审流程。

角色包括：

查看者。
设计者。
开发者。
评审者。
管理员。

协议状态包括：

草稿。
待评审。
已批准。

第一版可以只做状态标记，不强制实现完整审批流。后续如果工具进入团队协作场景，再加入多人评审、变更审批、发布记录和权限控制。

### 7.16 API 与持久化接口

前端不应直接操作零散文件，而应通过稳定业务接口访问工作区、协议 IR、网络地图和派生视图。业务接口可以由本地 HTTP 服务、Electron IPC 封装或两者兼容实现，但接口语义不应依赖 Electron。

核心接口包括：

workspace/open。
workspace/scan。
workspace/status。
protocol/get。
protocol/update。
protocol/generate。
protocol/layout。
protocol/lint。
protocol/diff。
network/get。
network/updateNode。
network/updateLink。
network/updateBinding。
network/topology。
network/estimate。
view/list。
view/save。
sync/status。
sync/resolve。

所有写操作都应遵循临时文件加原子替换，并记录输入文件 hash、IR 版本、网络模型版本和保存时间。解析失败、生成失败或 lint 阻断时，不应覆盖用户源文件。API 错误统一返回错误码、用户可读消息、诊断列表和可选源码位置。

## 8. 前端界面设计

ProtoVault 的界面风格参考 Obsidian，但内容面向协议工程。

### 8.1 左侧导航区

左侧应采用 Obsidian/IDE 式的双层结构：最左侧是窄工作栏，负责切换工作模式；其右侧是当前模式对应的树、列表或过滤器。左右栏和顶部工具栏都应支持拖拽调整、折叠和一键隐藏。

最左侧工作栏建议包含：

协议工作区。
关系图谱。
网络地图。
问题和诊断。
搜索。
设置。

协议工作区模式下，左侧内容区支持多种视图切换：

文件夹视图。
模块视图。
协议结构视图。
协议网络绑定视图。
依赖关系视图。
最近修改视图。
风险等级视图。
数据量大小视图。

文件夹视图对应真实 header 目录。模块视图对应分系统和模块划分。协议结构视图对应 struct、enum、字段和 header 等对象。风险等级视图用于快速查看存在质量问题或兼容性风险的协议。

网络地图模式下，左侧内容区不建议再做复杂树形编辑，而是提供节点类型、链路类型、协议类型、风险等级和业务视图的过滤器。真正的节点、链路和协议绑定编辑应放在中间区域的表格中完成。

### 8.2 中间编辑区

中间区域采用多标签页设计。用户可以同时打开多个协议结构、文档、关系图或代码视图。

协议编辑主要采用表格和表单方式。字段以表格形式展示，用户可以编辑字段名、类型、数组长度、单位、描述、默认值、版本状态和归档属性。

高级用户可以切换到代码视图查看生成结果。

多标签页应采用类似 Visual Studio 的预览标签和固定标签机制。单击树节点打开临时预览标签，双击或编辑后固定为正式标签，减少大量浏览造成的标签堆积。所有可编辑标签都应有脏状态标记，支持 Ctrl+S 保存，关闭或切换关键上下文时提示保存。

网络地图模式下，中间区域建议采用多标签页或分段表格：

节点表。
链路表。
协议绑定表。
业务视图表。
拓扑预览。

节点、链路和协议绑定以表格为主要编辑方式。拓扑预览负责展示和跳转，不作为第一版主要编辑入口。这样可以避免在复杂图谱中拖线改配置带来的误操作，同时保留图谱的空间理解价值。

### 8.3 右侧属性面板

右侧属性面板展示当前选中协议、字段、网络节点、通信链路或协议绑定的详细属性。

对于协议结构体，应展示：

padding 信息。

版本状态。
质量检查结果。
AI 建议。

对于字段，应展示：

字段类型。
字段偏移。
字段大小。
单位。
取值范围。
描述。
关联协议绑定数量。
关联链路摘要。
兼容性状态。

对于网络节点，应展示：

节点类型。
所属分系统。
部署位置。
软件运行时。
硬件画像。
发送协议数量。
接收协议数量。
估算发送数据量。
估算接收数据量。

对于网络链路，应展示：

源节点。

绑定协议数量。

用户点击问题后，可以跳转到对应协议、字段或配置项。

### 8.5 图谱视图

图谱视图用于展示协议之间、文件之间、网络节点之间和通信链路之间的关系。图谱应优先用于理解、筛选、定位和跳转，而不是承担第一版的复杂编辑。

第一版建议实现：

结构体依赖图。
include 关系图。
字段引用图。
协议网络拓扑预览。

后续扩展：

业务数据流派生图。
关键链路图。
版本演进图。
设备协议映射图。
性能瓶颈图。

图谱交互应做到：

滚轮缩放平滑。
拖拽平移流畅。
节点标签清晰。
不同节点类型颜色明确。
不同链路方向和含义有清晰高亮。
点击节点时区分入向链路和出向链路。
双击节点可跳转到左侧树和打开对应标签页。
主题颜色跟随全局主题，而不是图谱自带一套割裂主题。

### 8.6 主题、布局和可操作性原则

ProtoVault 的主题系统应兼容 Obsidian 风格的 CSS 变量，并在此基础上提供内置主题。主题切换入口建议放在左侧底部工作空间设置中，而不是散落在各个视图内部。

建议内置主题包括：

默认深色主题。
Obsidian 浅色主题。
Tokyo Night 兼容主题。
简墨中国风主题。

主题必须覆盖全局界面、字段表格、代码视图、属性面板、问题面板和关系图谱。图谱节点、连线、标签、选中态、入向链路和出向链路都应从统一主题变量中取色，避免出现表格黑底、图谱模糊或主题割裂的问题。

布局原则包括：

左栏、右栏和顶部工具栏都支持收起。
三栏宽度支持拖拽调整。
树图区域和底部工作区选择栏是两个独立滚动区域。
长树图滚动时不能遮挡底部工作区栏。
表格编辑、F2 编辑、右键菜单、双击打开、Ctrl+S 保存和脏状态提示保持一致。
右侧属性面板以展示为主，不承担主要编辑入口。
源码错误时仍允许用户进入 Header 源码标签修复，并通过诊断定位错误行。

## 9. 推荐 MVP 范围

提供基础语义 diff。

第二阶段增加：

## 10. 关键风险和边界

### 10.1 不要把第一版做成万能 C++ 解析器

C++ header 复杂度很高。如果第一版试图支持所有宏、模板、继承和条件编译，项目复杂度会迅速失控。

应明确支持协议 C++ 子集，并对不支持的结构给出提示。

### 10.2 不要让 header 文本成为唯一主数据源

如果前端直接编辑 header，后端又反向解析 header，容易出现双向同步问题。

更稳妥的方式是：

已有 header 导入为协议 IR。
前端编辑协议 IR。
工具从协议 IR 生成 header。

### 10.3 不要只做文件级 Git 版本管理

维护人员真正关心的是协议语义变化，而不是某几行代码变化。工具必须提供字段级、结构体级、协议绑定级和链路级的语义 diff。

### 10.4 不要让 AI 替代规则校验

AI 可以生成描述、解释风险和总结变更，但不能直接决定协议是否正确，也不能作为兼容性判断的唯一依据。

### 10.5 必须区分内存布局和序列化布局

C++ sizeof 结果和网络发送大小不一定相同。工具必须区分内存布局大小、序列化大小和实际运行数据量。

### 10.6 必须结合频率理解性能风险

协议大小本身不能直接等价于性能压力。一个大协议低频发送可能没有问题，一个小协议超高频发送可能成为瓶颈。数据量估算必须结合协议绑定中的发送频率、批量大小、链路传输方式、节点画像和业务场景。

### 10.7 不要把数据流作为主编辑事实

大型仿真和分布式系统往往存在多条业务数据流，但这些数据流共享同一张节点和链路网络。如果把每条数据流都作为独立主数据维护，系统会快速变散，且容易出现重复配置和互相矛盾。

更合理的方式是：

实体节点是主事实。
通信链路是主事实。
协议绑定是主事实。
业务数据流是派生视图。
AI 总结和瓶颈分析是派生结论。

当用户需要表达某个业务数据流时，应通过过滤、标签、视图规则或智能体总结从网络事实中提炼，而不是复制一套新的拓扑。

## 11. 项目价值

对于项目管理人员，ProtoVault 可以把协议规模、节点边界、链路关系、数据流向和变更影响量化展示，减少沟通依赖和决策盲区。

对于设计人员，ProtoVault 可以帮助建立清晰的数据契约，明确字段语义、单位、坐标系、时间基准和协议边界。

对于开发人员，ProtoVault 可以帮助定位协议布局、字段偏移、依赖关系、链路吞吐、节点瓶颈和代码生成问题。

对于维护和二次开发人员，ProtoVault 可以提供协议版本、变更记录、影响分析、可视化编辑和文档生成能力，降低接手成本。

## 12. 总结定位

ProtoVault 不是一个简单的 header 编辑器，而是一个面向分布式软件项目的数据协议资产管理工作台。

它以 Obsidian 类似的工作卷界面为入口，以 C++ header 扫描和协议 IR 建模为核心，以协议网络地图连接实体节点、通信链路和协议载荷。节点和链路是系统事实，协议绑定是数据传输事实，业务数据流和性能瓶颈是派生视图。工具通过可视化编辑、字节布局分析、版本治理、代码生成、依赖影响分析和本地 AI 辅助，最终帮助团队把数据协议从代码细节变成可管理、可理解、可分析的工程资产。
