# ProtoVault 版本资产与多工作区合并设计

本文档定义 ProtoVault 中 Git 分支、Tag、协议工作区和多文件夹合并导出的业务模型。目标不是把 Git 命令简单搬进界面，而是把版本控制变成“协议资产治理”的一部分。

## 1. 设计目标

ProtoVault 的版本能力应解决四类问题：

1. 用户能看懂当前协议资产处在哪条线：实验分支、客户版本、发布分支或临时修复线。
2. 用户能把某个稳定点固定为协议基线 Tag，用于交付、回归、Diff 和文档归档。
3. 用户能把多个文件夹或工作区作为协议包挂载、合并预览和导出，而不是依赖手动复制 Header。
4. 用户能在不破坏源文件的前提下查看冲突、差异和合成结果。

## 2. 核心心智模型

### 2.1 Branch：协议演化线

分支不是单纯代码分支，而是协议演化线：

- `main`：默认稳定线。
- `release/*`：面向交付或客户版本的稳定线。
- `feature/*`：协议实验线。
- `scenario/*`：仿真场景线。
- `hotfix/*`：紧急兼容修复线。

前端展示重点：

- 当前分支名称。
- HEAD 提交点。
- 是否存在未提交修改。
- 分支切换入口。
- 基于当前状态新建并切换分支。

### 2.2 Tag：可交付协议基线

Tag 表示“此时此刻的协议资产可以被引用和复现”。

推荐命名：

- `protovault/baseline/YYYYMMDDTHHMMSSZ`
- `release/v1.2.0`
- `customer/acme/radar-2026q3`

创建基线 Tag 的前置条件：

- 工作区必须在 Git 仓库中。
- 工作区必须干净。
- 已完成扫描、Lint 或必要的人工检查。

Tag 的用途：

- 与当前工作区做语义 Diff。
- 生成协议文档和报告。
- 作为多工作区合并时的输入版本。
- 作为导出协议包的版本标识。

### 2.3 Protocol Package：协议包

一个协议包是一个可独立扫描、索引、导出和版本化的协议文件夹。

最小字段：

- `id`
- `name`
- `rootPath`
- `gitRepositoryRoot`
- `branch`
- `commit`
- `tag`
- `headerCount`
- `typeCount`
- `diagnosticCount`
- `dirty`

当前 MVP 中，打开的工作区就是唯一协议包。后续多文件夹模式中，用户可以挂载多个协议包。

### 2.4 Composition：协议合成

多文件夹“合并”默认不是 Git merge，而是协议资产合成。

合成策略：

1. `partition`：按 namespace / 文件夹来源分区共存，适合多个子系统协议拼装。
2. `overlay`：按稳定 ID 或完全限定名覆盖，适合补充元数据、修正字段或组合客户差异。
3. `baseline-compare`：只做差异比对，不生成合成结果。

冲突原则：

- 同一稳定 ID 的类型出现不兼容结构变化时生成冲突报告。
- 同名类型但无共同稳定 ID 时提示人工确认。
- 字段顺序、类型、数组长度、枚举值、布局变化均进入语义变化记录。
- 合成结果默认写入 `.protocol/generated/` 或导出目录，不直接覆盖源 Header。

## 3. 前端布局方案

### 3.1 左侧 Rail

保留独立“源代码管理 / Git”入口。

进入该入口后：

- 左侧导航栏成为 Source Control + Version Vault。
- 中间区域用于 Diff、版本工作台或导出结果。
- 右侧 Inspector 展示 Git 摘要、当前基线、选中提交详情。

### 3.2 Source Control 左栏结构

推荐顺序：

1. 工作区摘要。
2. 暂存、取消暂存、基线 Tag、版本 Diff 快捷操作。
3. Commit 输入框。
4. Version Vault：
   - 当前分支 / HEAD / 最近 Tag / 协议包规模。
   - 分支切换和新建分支。
   - 最近 Tag 列表。
   - 多文件夹合并与导出设计入口。
5. Changes / Staged Changes。
6. Graph 历史流程图。

### 3.3 中间区域

没有打开 Diff 时，中间区域显示 Version Workspace 空白态：

- 解释分支、Tag、协议包的意义。
- 引导用户从左侧进行 Diff、基线、导出和分支操作。

打开 Diff 时：

- Working Tree Diff：未暂存内容对比。
- Index Diff：暂存区对比。
- Commit Diff：历史提交对比。

后续多协议包模式中，中间区域新增：

- 合并预览表。
- 冲突表。
- 导出向导。

## 4. 关键工作流

### 4.1 创建实验线

1. 打开工作区。
2. 进入源代码管理。
3. 在 Version Vault 中输入 `feature/protocol-xxx`。
4. 点击“新建并切换”。
5. 进行协议编辑、扫描、Lint、Diff。
6. 提交更改。

### 4.2 固定基线

1. 保证工作区干净。
2. 点击“创建基线 Tag”。
3. 输入推荐 Tag 名或自定义 Tag。
4. 生成 `.protocol/reports/` 下的基线摘要。
5. 后续可用该 Tag 做语义 Diff。

### 4.3 多文件夹合成

后续实现：

1. 点击“挂载文件夹”。
2. 选择多个协议根目录。
3. 系统扫描每个协议包，读取其 Git 分支和 Tag。
4. 用户选择合成策略。
5. 系统生成合成 IR、冲突报告和预览。
6. 用户选择导出文档、协议包快照或生成 Header。

### 4.4 导出版本包

后续实现的导出包建议结构：

```text
protovault-package/
  manifest.json
  headers/
  .protocol/
    workspace.json
    ir/
    meta/
    reports/
  git/
    branch.txt
    commit.txt
    tag.txt
  docs/
    protocol.md
    semantic-diff.md
    lint-report.md
```

## 5. 当前已落地 MVP

- Source Control 左栏保留 VS Code 式 Changes、Commit、Graph。
- 新增 Version Vault 面板：
  - 当前分支。
  - HEAD。
  - 最近 Tag。
  - 协议包规模。
  - 分支切换。
  - 新建并切换分支。
  - 最近 Tag 列表。
  - 创建基线 Tag。
  - 版本 Diff。
  - 导出协议文档。
- 中间 Git 空白态改为 Version Workspace。
- 多文件夹合并与导出已在 UI 中作为设计态入口展示，但不会执行破坏性操作。

## 6. 后续实现边界

下一阶段建议分三步：

1. **多协议包挂载**
   - Electron 多目录选择。
   - `.protocol/packages.json` 记录挂载列表。
   - 为每个包单独维护 scan status 和 Git status。

2. **合成与冲突引擎**
   - 以稳定 ID、namespace、文件路径建立类型索引。
   - 支持 `partition`、`overlay`、`baseline-compare`。
   - 生成冲突报告，不直接覆盖源文件。

3. **导出与切换**
   - 导出文档包。
   - 导出协议快照包。
   - 导出指定分支 / Tag 的协议包。
   - 支持在前端快速切换协议包、分支和基线。
